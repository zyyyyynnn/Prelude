## 一、核心引擎层的容错降级

- 实现了基于正则表达式柔性降级的 LLM 结构化提取。评估报告中的三维评分正则（`TECHNICAL_SCORE_PATTERN` / `EXPRESSION_SCORE_PATTERN` / `LOGIC_SCORE_PATTERN`）支持可选 Markdown 加粗 `**` 与可选 `分` 字后缀，兼容 LLM 输出格式波动。
- 单项评分解析失败时不再抛出 `BusinessException` 导致全局 500，而是记录 `log.warn` 并返回 `null`，实现柔性降级，保障核心面板不白屏。
- `stripJsonFence` 方法增加了兜底逻辑：当 LLM 在 JSON 前输出推理文字时，自动截取第一个完整的 `[...]` 数组，彻底解决 Markdown 标签干扰与 JSON 反序列化崩溃问题。

**对应代码**：`InterviewServiceImpl.java` — `TECHNICAL_SCORE_PATTERN`（第 72 行）、`extractScore`（第 521 行）、`stripJsonFence`（第 698 行）

## 二、文件流与内存安全（OOM 防范）

- 建立了 PDF 魔数（Magic Bytes）深度校验防线：`validatePdfMagicBytes` 方法读取文件头 4 字节，校验 `%PDF`（`0x25 0x50 0x44 0x46`）签名，阻断伪造扩展名攻击。
- 引入超大文件前置拦截：`MAX_PDF_PAGES = 50` 页数上限与 `MAX_FILE_SIZE = 10MB` 大小上限双重防护，防止恶意多页 PDF 耗尽 CPU。
- 存库文本安全截断：`MAX_RAW_TEXT_STORE_LENGTH = 100_000` 限制 MySQL 写入长度，并实现 `safeTruncate` 方法，通过 `Character.isHighSurrogate` / `Character.isLowSurrogate` 检测，防止 4 字节 Emoji 的 Surrogate Pair 被从中切断导致 utf8mb4 写入崩溃。

**对应代码**：`ResumeServiceImpl.java` — `validatePdfMagicBytes`（第 188 行）、`extractPdfText`（第 124 行）、`safeTruncate`（第 164 行）

## 三、闭环的安全与鉴权防线

- 实现底层 JWT 的防抖校验：`isRedirectingToLogin` 模块级状态锁防止多个并发 401 请求触发冗余路由跳转与状态清理。
- 并发 401 路由死锁拦截：`router.replace` 调用包裹在 `try...catch` 中，防止 Vue Router 抛出 `NavigationDuplicated` 或守卫拦截异常导致死锁锁永远无法释放。
- API Key 采用 AES-GCM 动态 IV 随机加密，前端通过 `apiKeyMasked` 字段展示脱敏值。
- 防误触覆写机制：前端保存时 `apiKey: apiKeyInput.value === '' ? undefined : apiKeyInput.value`，空字符串不传入后端，利用 Axios 自动剔除 `undefined` 的特性触发后端 `!= null` 防空值保护。
- 密钥清除通道：通过 `__CLEAR__` 占位符实现主动清空 API Key，后端识别该哨明值后将加密字段置为 `null`。

**对应代码**：`http.ts`（第 34 行 `isRedirectingToLogin`、第 65 行 `try...catch`）、`LlmSettingsView.vue`（第 85 行 `apiKey` 参数）、`UserLlmConfigServiceImpl.java`（第 50 行 `__CLEAR__` 判断）

## 四、生产环境与性能压榨

- 深度重构底层查询，消除 N+1 慢查询：`listCurrentUserResumes` 方法从逐简历 `selectCount` 改为基于 `QueryWrapper` 的 `GROUP BY` 聚合查询，通过 SQL 别名 `resume_id AS resumeId` 统一字段命名，使用强类型 `for` 循环遍历 `List<Map<String, Object>>`，彻底防住 NPE 和大小写变换陷阱。
- 补齐高频业务查询所需的复合索引：`idx_session_user_created`、`idx_score_user_created`、`idx_weakness_user_created`、`idx_resume_user_created`，均覆盖 `user_id + created_at` 组合，优化 `WHERE user_id + ORDER BY created_at` 查询路径。
- 前端 Vite 构建应用了精准的 `manualChunks` 拆包策略：`vendor-element-plus`、`vendor-echarts`、`vendor-zrender`、`vendor-brand`、`vendor-markdown-it` 五个独立 chunk，降低首屏加载体积。
- TypeScript 严格模式下修复了 `LlmConfigPayload.apiKey` 类型声明，从 `string` 改为 `string | undefined`，彻底消除编译期隐患。
- 创建独立的 `application-prod.yml` 生产环境配置：`spring.sql.init.mode: never` 防止意外重建表结构，`app.cors.allowed-origins` 限制跨域来源，`logging.level.root: warn` 收紧日志级别。

**对应代码**：`ResumeServiceImpl.java`（第 79 行聚合查询）、`schema.sql`（末尾 4 条 `CREATE INDEX`）、`vite.config.ts`（第 16 行 `manualChunks`）、`contracts.ts`（第 111 行 `apiKey?: string`）、`application-prod.yml`
