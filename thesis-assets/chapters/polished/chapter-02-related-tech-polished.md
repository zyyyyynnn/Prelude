# 第二章 相关技术综述

## 2.1 大语言模型与 OpenAI-compatible 调用接口

大语言模型（Large Language Models, LLMs）是近年来自然语言处理领域的重要技术基础。与传统基于规则或模板的文本处理方式相比，大语言模型在文本理解、上下文组织、自然语言生成和结构化信息提取方面具备较强的通用能力，能够基于输入语境生成相对连贯的文本结果。对于求职训练场景而言，大语言模型可用于理解简历内容、生成模拟面试追问、整理面试过程，并输出面试反馈报告[9-12]。

在本系统中，大语言模型主要承担四类任务：一是辅助简历结构化分析，将 PDF 简历中提取出的非结构化文本转换为技能、项目经历等字段；二是根据岗位模板、简历内容和历史问答生成模拟面试追问；三是在面试结束后根据完整对话生成总结性报告；四是配合 Structured Output 思路输出结构化内容，便于后端解析和前端展示。大语言模型由此成为连接简历诊断、模拟问答和反馈报告的重要能力来源。

为降低系统与特定模型厂商之间的耦合，本系统采用 OpenAI-compatible 调用方式组织模型请求。OpenAI-compatible 通常指一类在接口路径、请求格式或响应结构上参考 OpenAI API 风格的服务接口。该方式有利于在工程层面通过相对统一的调用网关接入不同兼容服务。需说明的是，不同服务在鉴权方式、模型列表接口、错误返回格式、流式输出格式和上下文长度限制等方面仍可能存在差异，因此不能直接认为所有 OpenAI-compatible 服务都具有完全一致的行为。

系统进一步支持用户级 BYOK（Bring Your Own Key）配置。用户可以在系统中配置 endpoint root、API Key 和运行模型。API Key 由后端加密保存，前端查询配置时不回显明文。用户保存配置后，聊天、流式面试和异步报告生成链路可复用该用户级模型配置。该能力不绑定默认厂商，也不推荐具体模型，而是提供可配置的模型接入方式。

大语言模型输出具有概率性和开放性，在实际应用中可能出现事实偏差、格式不稳定或评价边界不清等问题。因此，本系统将模型输出定位为求职训练辅助反馈，不将其作为真实招聘判断依据。在工程实现中，需要结合 Structured Output、字段校验、异常处理和边界约束等机制，降低模型输出不稳定对系统使用的影响。

## 2.2 Server-Sent Events 流式通信技术

Server-Sent Events（SSE）是一种基于 HTTP 的服务端到客户端单向事件流推送机制。客户端建立连接后，服务端可以持续向客户端发送事件数据，客户端根据事件内容进行增量处理。与传统轮询相比，SSE 可以减少客户端反复发起请求的开销；与 WebSocket 相比，SSE 更适合服务端持续下发、客户端被动接收的单向文本流场景。

在大语言模型应用中，模型通常可以按 token 或文本片段逐步返回生成结果。如果后端等待完整结果生成后再一次性返回，用户需要等待较长时间才能看到反馈。通过 SSE，服务端可将模型增量输出持续推送给前端，使用户在生成过程中逐步看到回答内容。因此，SSE 适合用于模拟面试中面试官回复的流式展示。

在本系统中，SSE 主要用于两类场景：一类是模拟面试过程中，后端将大语言模型返回的增量文本分片发送给前端，由前端逐步展示面试官回复；另一类是报告生成完成后，后端通过 SSE 广播 `report_ready` 事件，通知前端刷新报告内容。前者用于提升问答过程的交互连续性，后者用于配合异步报告生成链路完成状态通知。

前端在处理 SSE 或流式响应时，可以结合 `fetch`、`ReadableStream`、`TextDecoder` 等机制读取增量字节流，并将其转换为文本片段。对于高频文本片段，前端可使用 `requestAnimationFrame` 进行缓冲刷新，以减少频繁响应式更新带来的页面压力。在用户主动结束流程或切换页面时，可通过 `AbortController` 中止当前请求，减少多个异步请求同时修改页面状态的风险。状态核对和指数退避重试也可用于降低异常断连后的界面状态错乱风险。

需说明的是，上述机制属于流式交互的基础稳定性设计，用于支撑本地和演示环境下的功能链路。它们并不表示系统已经完成大规模长连接测试，也不意味着前端展示能够在所有环境中保持固定帧率。

## 2.3 Vue 3、TypeScript 与 Vite 前端技术

Vue 3 是一种用于构建用户界面的渐进式前端框架，适合开发单页应用和组件化页面。其响应式系统和组件机制能够将页面拆分为相对独立的功能单元，便于维护登录页、简历管理页、面试页、报告页和设置页等不同界面。Composition API 提供了更灵活的状态组织方式，适合在复杂页面中管理异步请求、表单状态、流式文本展示和组件复用逻辑。

TypeScript 是 JavaScript 的类型扩展语言，能够在开发阶段提供类型检查和接口约束。对于前端工程而言，TypeScript 有助于明确接口请求参数、响应数据结构、页面状态对象和组件属性类型，减少由于字段拼写错误或数据结构变化导致的运行时问题。在本系统中，TypeScript 可用于约束用户信息、简历对象、面试会话、消息记录、报告结构和模型配置等数据类型。

Vite 是现代前端构建工具，主要用于开发环境启动、模块热更新和生产构建。相比传统构建方式，Vite 在本地开发中具有启动快、反馈及时的特点，适合前端页面频繁调试的开发流程。本系统使用 Vite 支撑前端开发和构建，使登录、简历上传、模拟面试、报告展示和配置页面可以在本地环境下进行较为便利的调试。

在本系统中，前端主要承担以下职责：用户登录与路由控制、PDF 简历上传与解析结果展示、岗位选择与面试会话创建、SSE 流式文本展示、报告页面渲染、数据看板展示以及 BYOK 模型配置管理。前端通过 HTTP 接口与后端服务交互，并根据后端返回的状态更新页面。对于流式问答和异步报告生成等场景，前端还需要处理请求中止、状态核对和事件通知等交互逻辑。

Vue 3、TypeScript 与 Vite 的组合能够满足本系统前端页面开发和本地调试需求，其作用主要体现在提高开发组织性和可维护性，并不意味着这是唯一实现路径，也不代表特定交付等级。

## 2.4 Spring Boot 与后端服务架构

Spring Boot 是 Java 生态中常用的后端应用开发框架，能够简化 Web 服务、依赖管理、配置加载和中间件集成等工作。通过 Spring MVC，后端可以对外提供 REST API；通过拦截器、过滤器和统一异常处理机制，可以实现身份校验、参数检查和错误响应封装。对于本系统而言，Spring Boot 是后端业务流程组织的基础。

本系统后端主要负责用户认证、简历解析、岗位模板管理、面试会话管理、消息记录、模型调用、报告生成、中间件集成和数据持久化等功能。用户登录后，后端通过 JWT 实现接口鉴权，并在请求处理过程中维护当前用户上下文。简历、面试会话和报告等资源都需要根据当前用户进行归属校验，以减少越权访问风险。

MyBatis-Plus 是基于 MyBatis 的增强工具，可简化实体映射、通用 CRUD 操作和条件查询构造。本系统使用 MyBatis-Plus 连接 MySQL 数据库，并将用户、简历、岗位模板、面试会话、面试消息、报告和统计数据等对象持久化。MySQL 作为关系型数据库，适合保存具有明确结构和关联关系的业务数据。

在后端服务架构中，Redis、RabbitMQ 和 LLM 调用网关分别承担不同职责。Redis 主要用于限流、缓存、评分锁和状态辅助；RabbitMQ 用于报告生成异步任务队列；LLM 调用网关用于封装模型请求、流式响应处理和用户级模型配置读取。后端还需要在长耗时模型调用、异步报告生成和流式通信中进行状态校验和异常处理，以保证业务流程在可控范围内运行。

Spring Boot 的作用在于为上述业务模块提供统一的服务组织方式。本文仅在相关技术层面说明其适用性，不将其表述为特定规模部署能力或性能验证结论。

## 2.5 Apache PDFBox 文档解析技术

Apache PDFBox 是 Java 生态中常用的 PDF 文档处理库，可用于读取 PDF 文档、提取文本、访问页面信息和处理部分 PDF 元数据。在求职训练系统中，PDF 简历是用户输入的重要材料。系统需要从 PDF 中提取文本内容，供后续简历结构化分析、岗位匹配和面试问题生成使用。

PDF 文档内部结构较为复杂。不同简历模板在字体、段落、表格、多栏排版和文本对象顺序等方面可能差异较大。对于扫描件或图片型 PDF，如果文档中不存在可直接提取的文本层，仅依靠 PDFBox 可能无法得到有效文本。因此，PDF 文本提取结果需要结合异常处理和后续校验，不能认为所有 PDF 简历都能被准确解析。

在本系统中，PDFBox 用于对用户上传的 PDF 简历进行文本提取。为了降低异常文件对系统的影响，系统需要在解析前进行文件类型校验、PDF 魔数校验、文件大小限制和基础异常处理。提取后的文本还需要进行长度限制和空文本判断，避免无效或过长内容直接进入模型调用流程。

PDFBox 提取结果是后续大语言模型结构化分析的输入。模型可在提取文本基础上识别技能、项目经历等字段，但最终效果仍受 PDF 排版质量、文本提取完整性和模型输出稳定性影响。因此，PDFBox 在系统中承担基础文本提取职责，而不是完整的简历理解和评价能力。

## 2.6 Redis 缓存、限流与状态辅助

Redis 是常用的内存数据存储组件，具有读写速度快、数据结构丰富和部署简单等特点。它可用于缓存、限流、临时状态保存、分布式辅助控制等场景。在 Web 应用中，Redis 常用于降低数据库访问压力、保存短期状态或实现简单的请求频率控制。

在本系统中，Redis 的主要职责包括 LLM 调用限流、缓存、评分锁、演示状态辅助和局部状态辅助。由于大语言模型调用通常具有成本和外部服务依赖，系统需要对短时间内的重复请求进行一定约束。Redis 可结合计数器、过期时间或 Lua 脚本实现基础限流逻辑，减少重复调用对系统和外部服务造成的压力。

Redis 还可用于评分锁或局部状态辅助。例如，在需要避免重复操作的场景中，可以通过 Redis 的原子操作实现短期锁控制；在 Demo Twin 演示或本地测试流程中，也可以使用 Redis 保存部分临时状态。上述用途均属于缓存和状态辅助范围。

需明确的是，当前系统中 Redis 不再承担报告生成任务队列职责。报告生成异步任务由 RabbitMQ 承担。Redis 不用于报告任务投递，也不用于保证报告任务的可靠处理。Redis 限流机制属于基础保护设计，不能等同于压力测试结论。

## 2.7 RabbitMQ 消息队列技术

RabbitMQ 是基于 AMQP 协议的消息队列中间件，常用于系统间异步通信、任务转交和业务解耦。生产者将消息发送到交换机或队列，消费者从队列中获取消息并执行相应处理。通过消息队列，可以将耗时任务从用户请求链路中拆分出来，降低同步等待时间，并使任务处理过程相对独立。

在本系统中，RabbitMQ 用于报告生成异步任务队列。面试报告生成通常需要读取历史对话、构造提示词、调用大语言模型并解析结果，耗时受模型服务和网络环境影响。如果在 `/finish` 请求中同步等待报告生成完成，容易造成请求阻塞。因此，系统通过 RabbitMQ 将报告生成任务交由后端消费者异步处理。

报告生成链路为：

```text
/finish → generating → RabbitMQ → ReportJobWorker → summary_report → finished → report_ready
```

用户触发 `/finish` 后，后端校验会话归属和状态，将会话状态更新为 `generating`，生成 `jobId`，并发布报告任务消息。此时 `/finish` 返回 `generating + jobId`，仅表示报告任务已进入异步生成链路，不表示报告已经完成。`ReportJobWorker` 通过 `@RabbitListener` 消费 RabbitMQ 消息，读取会话数据并生成报告。报告生成后，系统将结果写入 `summary_report`，将会话状态更新为 `finished`，并通过 SSE 广播 `report_ready` 事件，通知前端刷新报告。

RabbitMQ 的引入主要用于解耦长耗时报告生成与用户结束面试请求，使报告生成流程具备更清晰的异步处理边界。当前系统尚未引入 DLQ、outbox、publisher confirm、消费并发调优等可靠消息增强机制，因此不能将其表述为完整的可靠消息处理方案。后续若系统面向更复杂的部署环境，还需要补充消息确认、异常补偿和任务恢复等机制。

## 2.8 Structured Output 与结构化报告解析

Structured Output 是指通过提示词、结构约束或模型接口能力，使大语言模型尽量按照预期格式输出内容。在需要后端程序解析模型结果的场景中，纯自由文本可能导致字段缺失、格式不一致或解析失败。通过 Structured Output 思路，可以减少自由文本带来的不确定性，使模型输出更接近后端可处理的数据结构。

在本系统中，Structured Output 主要用于面试报告生成结果解析。系统希望从模型输出中获得面试总结、能力评分、优势分析、薄弱点建议等字段，并将其用于报告展示和数据看板。如果模型返回完全自由的长文本，后端难以稳定提取评分和结构化建议。因此，系统在报告生成阶段对输出字段进行约束，并在后端进行反序列化和字段校验。

需要注意的是，Structured Output 只能降低解析失败风险，不能保证模型输出始终合法。后端仍需对解析结果进行字段校验、分数范围约束和异常容错。例如，当某些字段缺失、分数超出范围或结构不符合预期时，系统应进行兜底处理，避免异常内容直接影响前端展示。

报告评分和建议应定位为求职训练辅助反馈。由于模型输出受到简历质量、对话内容、提示词和模型能力影响，报告结果不能作为求职者能力的最终判断，也不能替代真实招聘流程中的人工评价。

## 2.9 本章小结

本章围绕智能模拟面试与简历诊断系统涉及的主要技术进行了综述。大语言模型为简历理解、问题生成、对话组织和报告生成提供基础能力，但其输出存在不确定性，需要结合 Structured Output、字段校验和异常处理降低风险。OpenAI-compatible 调用方式与用户级 BYOK 配置为系统接入不同模型服务提供工程灵活性，但不意味着兼容所有平台。

SSE 支撑系统中的流式问答展示和报告完成事件通知，适合当前以服务端文本下行为主的交互场景。Vue 3、TypeScript 和 Vite 支撑前端页面开发与本地调试；Spring Boot、MyBatis-Plus 和 MySQL 支撑后端业务服务与数据持久化；PDFBox 用于 PDF 简历文本提取；Redis 用于限流、缓存、评分锁和状态辅助；RabbitMQ 用于报告生成异步任务队列；Structured Output 用于降低报告解析失败风险。

上述技术共同构成系统设计与实现的基础。下一章将在这些技术基础上，对系统需求、总体架构、数据模型和核心业务流程进行分析与设计。
