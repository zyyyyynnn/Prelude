# 第三章 系统分析与设计

## 3.1 功能性需求分析

本系统深度聚焦于高校毕业生的求职训练垂直场景，其核心业务工作流严格围绕“用户身份建立—简历自动化处理—面试会话初始化—多轮动态交互—量化结果评估反馈”这一主链路展开设计。基于系统顶层设计目标与各模块的内聚边界划分，本系统主要涵盖用户注册与安全登录、简历智能解析与诊断、模拟面试引擎触发、深层多轮流式对话以及多维评估报告生成等五大核心用例集群。这些核心用例的交互流转与协同关系如图 3.1 所示。

必须予以严肃声明与明确界定的是，本系统的工程与学术定位纯粹为“求职能力辅助训练与赋能工具”，绝不触及或试图干预企业侧真实的招聘录用决策网络。系统基于预训练模型与实时上下文衍生出的具有压迫感的面试追问，以及最终聚合生成的量化评估报告，其核心应用价值仅局限于辅助受训用户进行高频度的自我压力测试与认知复盘，在任何情况下均不应被视为对用户综合技术能力与职场潜力的最终、具有法理效力的定量裁决。这一极其审慎的系统边界锚定，不仅从源头上剥离了业务越权风险，更在工程实现上有效对冲并降低了大语言模型在复杂推演中可能诱发的逻辑幻觉（Hallucination）、内生数据偏见（Bias）以及生成评价体系非确定性所带来的一系列连锁不可控后果[9,12]。

图 3.1 系统核心用例图

在用户注册与登录这一前置用例中，核心参与者为匿名系统访客。其业务前置约束条件为：当前客户端未处于任何有效会话状态，且在注册场景下，所申请的唯一认证标识（如用户名）未被底表占用。该用例的底层流转逻辑为：用户在前端身份认证网关选择操作模式并提交结构化凭证表单；服务端拦截器在应用层对数据载荷进行严格的正则合法性校验，并在注册管线中通过索引查询执行唯一性防撞击检查；注册流程落库成功后，系统即完成用户实体记录的安全持久化；而在登录态确立后，服务端将通过加密签名算法生成具备严格时效与防伪造特征的 JWT（JSON Web Token）认证令牌，并随 HTTP 响应安全下发；前端接收后将令牌妥善驻留于本地存储结构中，从而打通引导用户进入后续核心鉴权路由页面的闭环通道。该用例的根本设计初衷在于为全域操作构建一个具备强一致性的底层身份追踪锚点，从而为后续错综复杂的简历挂载、隔离会话建立及历史评估溯源提供极其稳固的数据血缘根基。

进入简历上传与 AI 智能诊断这一核心数据拉取用例，其主导参与者变更为已持有合法令牌的认证用户。该链路的准入前置条件严苛设定为：用户身份认证器返回 PASS 状态，且目标数据包在 MIME 类型（严格限制为 PDF 规范）与字节大小上限均符合服务器 I/O 防御阈值。用户在前端投递 PDF 简历载荷后，后端应用引擎首先触发多层防篡改的文件特征校验机制，随后调度内部挂载的 PDF 物理版面解析组件执行高频的底层字符坐标剥离操作，以获取高纯净度的原始一维文本序列。紧随其后，系统将这批经过初步清洗的特征文本作为基座上下文，打包发送至外部配置的大语言模型微服务集群中，强制执行高维度的非结构化信息到结构化图谱（如硬技能图谱萃取、长周期项目经历解构等）的转化与提取作业。最终，系统开启数据库事务锁，将带有 MD5 签名的简历原件副本与清洗重构后的结构化特征数据一并落盘持久化，并将可视化的多维诊断矩阵回传至前端 DOM 树进行渲染。该用例在整体系统工程中扮演着决定性的前置引擎角色——它为后续即将展开的沉浸式模拟面试提供了海量且高度个性化的隐式语义背景，彻底确保了所有的动态问答追问均能够死死咬住用户真实的过往履历不放。

在触发并建立模拟面试引擎这一枢纽用例中，系统强制要求参与者不仅须处于高权限登录态，更必须已成功绑定至少一份结构化诊断完毕的有效简历资源。其业务触发门槛依赖于：用户已在前端沙盒中明确选定被激活的简历载荷，且系统后台已成功预热并挂载对应的多维度岗位知识模板。当用户在控制面板下达“开始面试”指令并选定目标狙击岗位后，后端引擎将立即在底层分配并创建一个具有绝对生命周期的面试会话主干记录实体，同时动态索引并提取该岗位模板中所深度绑定的、极具角色扮演约束力的系统级提示词（`system_prompt`）。随后，引擎将这份充当最高逻辑指令的提示词作为序列第一条系统强控消息（System Role Message），牢牢钉死在会话明细队列的头部；一切就绪后，后端下发该会话的唯一全局追溯标识（Session ID），并平滑引导客户端路由至沉浸式面试核心交互舱。本用例的架构使命，在于通过一系列精密的前置组装，将特定岗位的专业语境约束场、用户千人千面的履历特征向量与后续一切将要发生的动态多轮博弈，完美焊死在同一个拥有独立沙盒隔离的会话上下文中。

在极具挑战性的多轮对话面试核心用例中，交互的主体双方为处于高度专注状态的受训用户与由模型驱动的虚拟面试官。其运转的前置要求为：目标 Session ID 处于有效存活且未被主动熔断的 `IN_PROGRESS` 进行态。当用户在交互端完成当前轮次的思考并提交防御性陈述文本后，后端调度器将无延迟地将该条 `user` 角色消息落库锁定，并旋即启动逆向查询器，严格依据 `seq_num` 序列号从深层库表中将当前会话从创世之初至今的所有历史上下文片段进行升序拼装合并。随后，系统携带这批沉重的历史上下文负荷，跨越广域网向外部大语言模型推理集群发起高度不确定的流式推演调用；并在接收到模型响应的第一毫秒起，通过底层搭建的 SSE（Server-Sent Events）单向长连接隧道，将模型吐出的面试官追问片段如水流般源源不断地向前端渲染层进行高频推流。直至收到模型下发的终结截断符，系统方才执行收尾事务，将这段完整的虚拟面试官回复（`assistant` 角色）安全写入底表，系统在此进入静默监听态，蓄势等待用户的下一轮防守反击。该用例无疑是系统引擎中运转负荷最重、容错率最低的核心地带；其工程设计的心智重点，全部倾注于如何在维持长程上下文严格连贯、严防历史消息总 Token 数击穿模型容量上限的同时，依然能够利用流式解耦技术为用户营造出毫秒级响应、极致压迫且无比接近真实职场面试的沉浸式即时反馈体验。

在最终闭环的查看 AI 生成多维评估报告用例中，操作主体为已结束残酷推演过程的结业用户。其执行前置边界为：当前会话状态机已被手动或因超限而被系统强制拨转至结束态，且后端引擎拥有对该会话全生命周期内所有序列消息的完整读取权限。在用户触发“结束面试并复盘”指令后，后端后台守护进程将被唤醒，它会贪婪地吞吐当前会话中产生的所有对话明细，依照严格的评估工程模板组织出庞杂的诊断提示语树，并发起一次极为昂贵的宏观全局模型总结调用。随后，系统接收模型返回的结构化结案陈词，将其序列化后深埋入面试会话主记录表中，并同步将该会话状态彻底冻结。最终，客户端界面将依据这批高度结构化的 JSON 载荷，向用户全景渲染出包括但不限于岗位技术栈匹配度雷达图、语言表达逻辑清晰度量化打分、高光优势矩阵提取以及极其致命的薄弱点改进干预建议等深层指标。该用例的架构意义在于，彻底打破了传统面试“结束即黑盒”的痛点，通过将海量、碎片化且高度非结构化的多轮口语交互残局，最终沉淀、提纯为一份极具指导意义的结构化训练反馈蓝图，从而真正意义上帮助用户完成了从“参与受虐”到“认知觉醒”的一整套完整学习复盘周期。

## 3.2 非功能性需求分析

在确保上述功能流线运转无阻的基础之上，系统级架构必须向更为苛刻的非功能性质量约束指标（Non-Functional Requirements）妥协与挑战。在首当其冲的系统级响应时延控制（Latency Control）层面，本系统摒弃了传统 Web 架构下那套令人绝望的同步阻塞等待模式，全链路深度集成并采用了异步流式输出机制。这一架构级颠覆的根本目的，在于极限压缩并榨干用户从点击“发送”按钮那一刻起，到其肉眼在屏幕上捕捉到虚拟面试官吐出第一个有效字符反馈之间那极其漫长的感知等待时间。立足于当前的系统定位与网络基建水平，后端工程组为 SSE 通道下达了极其严苛的工程指令：将核心的“首字节响应到达时间（Time To First Byte, TTFB）”铁律般地压制在 2 秒的安全红线以内。这一非功能性指标的达成，彻底扭转了人机交互的心理预期，使整个模拟面试对抗过程彻底摆脱了“长时间白屏死锁后突然弹出万字长文”的机械感，进而极大程度地向毫秒级即时反馈的真人面试压迫感与沉浸感无缝贴近。

在关乎系统生死存亡的数据安全与隐私隔离（Data Security & Isolation）红线维度上，架构设计有着绝不容挑战的铁律：坚决杜绝将任何具有计费或越权调用风险的第三方底层大模型服务访问密钥（如 DeepSeek API Key 或 OpenAI Secret Token）以硬编码形式暴露给脆弱的前端编译包，或意外泄露并上传至公开的代码版本托管仓库中。鉴于此，系统核心密钥将永远被物理封印于受高度权限保护的后端生产环境本地环境变量配置文件之中，并强制要求前端所有的模型调用请求必须且只能通过系统自身部署的后端代理引擎池作为唯一受信任的反向代理跳板来发起间接路由访问。更为重要的是，系统内产生的海量高隐私价值资产——包括用户的登录凭证、高度敏感的原始个人简历附件、结构化履历画像以及长达数十轮的面试对抗历史推演明细，在数据库底层的任何一次 DML（数据操纵语言）越权读写操作，都必须接受拦截器层面基于 JWT 解码后的当前访问者身份（User ID）强绑定校验鉴权。此举在引擎深处彻底熔断了跨域越权（IDOR）及水平越权访问他人隐私数据的任何可能性灾难。

在探讨系统架构的高可用性与容灾逃生机制（High Availability & Fallback Mechanisms）时，必须直面一个极其残酷的客观现实：本系统在最核心的心智推理引擎环节，深度且重度地依赖于极不可控的外部异地大语言模型微服务集群。这就要求我们的工程架构必须充分预估并从容应对由于第三方接口遭遇突发限流、光缆抖动引发的长连接断流断连、或是由于输入负载包含触发审查机制敏感词而导致的异常熔断返回等一系列灾难性极端边缘情况（Edge Cases）。为此，系统在前后端均布设了极其厚重的防御性降级重试装甲：例如，在底层的 SSE 流式推流过程中一旦遭遇不可逆的网络异常中断，前端状态机将立刻拉响警报并向用户抛出极其清晰明了的底层断连阻断提示，并同步在底层暴露指数级退避重连（Exponential Backoff Retry）的恢复钩子；同时在面对偶发的模型格式化强输出解析失败等数据损坏（Data Corruption）场景时，系统允许并在容忍阈值内触发受限的自动重发兜底逻辑，以此从根本上确保客户端界面永远不会陷入那令人绝望的、死寂般不可逆的挂起等待死循环状态。

## 3.3 数据库设计说明

本系统在数据持久化基座层的底层关系架构设计，紧密围绕着用户实体（User）、简历资产（Resume）、岗位模板（Position Template）、面试会话主记录（Interview Session）以及微粒化的单点对话消息（Interview Message）这五大核心骨干对象展开深度拓扑建模。为进一步夯实数据在更高阶业务维度的聚合与溯源能力，架构师还针对性地补充引入了诸如对话推进阶段枚举（Stage）、多维评分历史快照（Score History）、用户能力薄弱点特征库（Weakness）以及兼容适配不同底层大语言模型供应商的调度配置表（Provider Config）等一系列支撑性对象组。这套高度内聚的范式化表结构设计，完美响应并支撑了系统在超大规模全景下的多租户身份隔离管理、长效数据安全复用、复杂逻辑上下文组织构建以及评估结果穿透追踪等深水区业务需求。系统底层关系型数据库的宏观实体联系图谱（E-R 图）布局如图 3.2 所示。其中，`user` 底表作为存储所有系统自然人注册用户基础画像信息的绝对权威数据源，是整个数据库联邦中所有派生业务数据的终极归属锚点；紧密挂载的 `resume` 表，则被专职用于安全托管用户历次上传的实体简历文件别名映射、未脱敏的原始全量文本数据以及经过重度清洗后的多维结构化特征萃取结果矩阵。在拓扑关系上，该表通过强一致性的外键约束 `user_id` 与主用户信息基表确立了标准的一对多级联关系，从而在物理架构上天然且健壮地支持同一活跃用户能够无缝并行管理、无损切换多份针对不同求职赛道的定制化简历档案库。

独立部署的 `position_template` 表则被赋予了极其特殊的架构使命，它用于静态存储并版本化管理系统内置的一系列高规格、强约束的特定岗位语境模板及其深度绑定、不可篡改的超级提示词（`system_prompt`）内容核心载荷。该表在设计哲学上的终极目的，在于将那些本应被硬编码在脆弱业务代码深处、且极难随着业务需求变动而灵活扩容的针对不同岗位的面试官冷酷人设设定、极其苛刻的特定专业提问风格偏好以及隐晦的综合考察评价重点，执行极其彻底的外化、剥离与抽象操作；从而在系统的业务主干之外，硬生生地开辟、形成了一个支持热插拔配置、具备极强横向扩展能力的高维岗位语义控制层。作为核心枢纽节点的 `interview_session` 表则扮演着统驭全局的面试会话总控司令部角色。它不仅肩负着精准记录发起该次残酷推演挑战的目标用户、其当时挂载的具体靶向简历实体以及所选择的严苛岗位模板之间那复杂的三角关联关系网，更需要时刻持久化跟踪并保存当前所挑战的目标岗位全称、会话主干有限状态机当前所处的生灭状态节点，以及在历经沧桑并最终全盘结束时由超级模型聚合生成的终局评估诊断报告全景内容。这一系列精密设计，为系统后续构建全域历史面试推演记录查询矩阵以及高精度的多维结果全链路追踪回溯，提供了一个高度内聚且无可辩驳的统一上帝视角查询入口点。

在整个关系型数据库的精密齿轮网络中，`interview_message` 明细表无疑是支撑那极其庞大、错综复杂且极易崩溃的多轮对话序列组织架构的终极基石表。在该张承担海量 I/O 并发的超大表中，`role` 字段作为极其关键的身份鉴权器，被用于严格界定并区分每一条消息序列的产生源头，其值域被极其克制地强制收敛于 `system`（系统上帝视角）、`user`（受训求职者）与 `assistant`（冷酷的虚拟面试官）这三种不可逾越的枚举角色标识中。具体细分而言，被标记为 `system` 角色的元首级消息，将独占并用于封存那决定整个面试走向的岗位模板超级提示词，并将其作为统御整个漫长会话周期的创世初始约束条件；而频繁产生的 `user` 消息实体则忠实记录了求职者在每一次高压逼问下被迫作出的文字输入反击；相对应的，`assistant` 消息则原封不动地沉淀了系统后台大语言模型经过深度推理后所生成的那些极其尖锐的面试官连环反问与无情质疑。与此同时，该表中存在一个同样具备决定性战略地位的隐式控制字段 `seq_num`（序列号锁）。该字段的唯一神圣使命，即在于极其严酷地锚定并记录下该条特定消息在所归属的那个孤立单体会话宇宙内部的绝对时序位置，从而在底层物理层面上死死捍卫了系统能够在将来任何时刻，均能以绝对严谨、无任何错乱时序的拓扑结构将漫长的多轮上下文原样重构拼装出的不可更改承诺。后端业务应用层在每次需要向该表追加入库哪怕一条微不足道的新消息实体时，都会极其严密地自动接管并递增维护该 `seq_num` 的自增连续性；且在后续引擎需要提取拼接这批极其昂贵的 `messages` 载荷数组用以前往外部大模型请求推演时，将雷打不动地依据该字段执行严格的升序提取操作。这套堪称偏执的设计，从根本上锁死了多轮对话那极易发生混沌翻转的上下文语境，确保了其在任何维度的拼装校验中均能展现出绝对稳定、坚不可摧的时序状态。此外，外围辅助挂载的 `interview_stage` 追踪表则被用于极其精准地测绘并记录面试流程在“破冰寒暄—底层技术深挖—高维压力推演—礼貌收尾总结”这一系列预设阶段中的真实推进演化情况；而 `score_history` 历史高频快照表与 `user_weakness` 弱点探针表则在暗处悄无声息地疯狂沉淀着每一轮激烈交锋中所产生的微小评分波动以及用户在无意识中暴露出的极其隐蔽的能力漏洞统计数据。正是依赖于 `role` 身份隔离与 `seq_num` 绝对时序这套天衣无缝的组合拳配合，才使得本系统能够在最底层的冷冰冰的数据库磁盘层面，极其完美地保留住了原本在内存中稍纵即逝的整个多轮口语对抗对话的完整高维拓扑结构，并从底座基础上为系统后续即将展开的一系列诸如极易失控的上下文滑动窗口安全截断、具有教学意义的复盘历史回放、终局深层报告自动化提炼生成以及微观维度的技术能力诊断分析等高难度逻辑，注入了无可比拟的底层赋能支撑。

## 3.4 系统架构说明

从宏观的上帝俯瞰视角审视整体系统架构，本系统在工程实现上极其标准且严密地构建了一套由前沿浏览器客户端展示层（Client Layer）、高并发的 Spring Boot 业务逻辑驱动服务层（Application Service Layer）、极其稳固的 MySQL 核心数据持久化存储层（Data Persistence Layer）以及挂载在云端的异构 DeepSeek/OpenAI 兼容协议认知模型服务层（LLM Cognitive Layer）这四大存在显著物理隔离与职责边界的独立层次所共同构筑的宏大拓扑模型。该系统的整体物理部署及逻辑流转总体架构图谱如图 3.3 所示。其中，驻留在浏览器端的前端客户端（Client Side）作为最前沿的接触探针，全权担负着与用户的零距离高维界面交互展示重任，包揽了复杂组件的渲染、大容量 PDF 简历的断点投递、高频打字机效果的面试激烈问答实时流式重绘，以及极其直观的结构化雷达图评估报告全景交互查看；部署在服务端的 Spring Boot 引擎组作为整个系统的大脑枢纽（Control Plane），则居中负责统御调度所有错综复杂的业务微服务编排、极其敏感的网关 JWT 权限硬校验拦截、针对高负载文件流的拆包处理、极其频繁的数据库底层读写并发锁控制，以及作为反向代理发起对外域云端高深认知大模型的安全越界调用；潜伏于底层的 MySQL 关系型数据库集群，作为系统绝不容有失的数据心脏，安全静默地存储并守护着包括海量活跃用户注册表、极其私密的简历档案池、生命周期极长的高维会话记录及上万条碎片化消息体等支撑系统生存的核心业务数据底座；而部署在广域网外部极具算力压迫感的大语言模型服务矩阵，则作为名副其实的外挂智能中枢（Brain Backend），全天候承接着包括从无序简历文本中极其艰难的提取结构化特征图谱、在极度错乱的面试语境中快速生成极具攻击性的动态问答回击，以及在庞杂混乱的数百条历史信息残局中冷静地总结输出全景评估报告等一系列最具挑战性的心智计算重任。

为确保在极度严苛、混合高低频次与各类长短生命周期交织的并发场景下，系统依然能够展现出极其顺滑且坚韧的运行姿态，系统在网络传输层内部极其创新性地规划并硬隔离出了两条完全物理剥离的独立主次通信通路管道。其一，是支撑常规业务流转的底层基础 RESTFul 短连接通路主干道。该通路全权负责承接并消化包括用户注册登录的瞬间鉴权、体积庞大的 PDF 简历一次性上传校验接收、海量且高并发的静态岗位模板列表探查查询、低频但必须绝对成功的会话实体开启创建操作，以及极其昂贵的历史长程记录全量提取查询与最终诊断报告的完整获取等经典请求。此类传统且循规蹈矩的请求数据包通常极其严格地遵循着“客户端主动发起叩门—服务端被动接管并阻塞处理—短时间内一次性打包返回最终结果断开”这一最为传统且古老的 HTTP 单次响应生命周期模式。其二，则是本系统引以为傲且专为应对极端流式推演而暴力开辟的、极其昂贵且极具技术难度的 SSE（Server-Sent Events）单向长连接专属超车道通路。这条极为特殊的管线，在绝大部分时间里保持死寂，其存在意义仅仅主要被用于应对模拟面试极速交锋过程中，那极其变态的流式动态问答渲染场景：当且仅当客户端在前端防抖隔离区以 `POST` 方式重锤提交包含极大 Token 负载的用户长篇防御回答包之后，服务端响应引擎将在接管该请求的瞬间，极其果断地将该任务扔进并放逐到彻底与 Tomcat 主请求线程绝缘的独立异步工作池深处；在此异步沙盒中，守护线程将向远端高维大语言模型云端网关发起极其漫长的流式探针接口调用，并利用建立于应用层底部的 `SseEmitter` 通信句柄，在长达几十秒甚至几分钟的漫长周期内，将模型缓慢吐出的、带有极强逻辑攻击性的碎片化生成字符内容片段，如同绵绵不绝的水流一般源源不断地跨越长城，持续向翘首以盼的前端浏览器客户端进行毫秒级的降维推送与界面增量重组渲染。正是通过这种在架构基建层面上，极其粗暴且有效地将短生命周期、高并发的基础常规数据 REST 请求管线，与极度长尾、占用资源极高、必须具备极低响应延迟的流式交互 SSE 长连接请求管线实施了物理与逻辑层面的双重彻底剥离切割机制；本系统才得以在面临不可预知的极端混合流量海啸冲击时，依然能够从容且优雅地同时完美兼顾传统业务处理底层逻辑架构那不可撼动的极高安全性与稳定性，以及在沉浸式高压面试交互前沿阵地那极度变态的毫秒级实时超低延迟苛刻指标要求。
