# 第三章 系统分析与设计

## 3.1 功能性需求分析

本系统面向求职训练场景，主要目标是提供简历解析、岗位匹配、模拟面试和报告反馈等辅助功能。系统定位为训练辅助工具，不替代真实招聘流程、HR 判断或录用决策。结合系统业务流程，功能性需求可归纳为用户认证、简历上传与解析、岗位匹配、多轮模拟面试、异步报告生成、用户级模型配置、报告展示与数据看板以及 dev fixture 本地验收辅助能力等内容。

### 3.1.1 用户注册与登录需求

用户认证是系统访问控制和数据隔离的基础。系统需要支持用户注册和登录，用户完成登录后通过 JWT 访问受保护接口。后端在处理简历、面试会话、报告和模型配置等数据时，应根据当前登录用户校验资源归属，避免用户访问或修改不属于自己的业务数据。

用户认证模块需要满足以下需求：

1. 支持用户注册、登录和登录状态保持。
2. 登录成功后向前端返回 JWT。
3. 前端在访问受保护接口时携带 JWT。
4. 后端通过拦截器或等价机制校验 JWT 有效性。
5. 后端根据当前用户身份对简历、会话、报告等资源进行归属校验。
6. 当用户未登录、令牌缺失或令牌无效时，系统应拒绝访问受保护接口。

### 3.1.2 简历上传与解析需求

简历是系统进行岗位匹配和模拟面试的重要输入。系统需要支持用户上传 PDF 格式简历，并对上传文件进行基础校验。后端接收文件后，使用 PDFBox 提取简历文本，再结合大语言模型对简历内容进行结构化分析，生成技能、项目经历等字段信息。

简历上传与解析模块需要满足以下需求：

1. 支持用户上传 PDF 简历。
2. 对上传文件进行文件类型、PDF 魔数、文件大小等基础校验。
3. 使用 PDFBox 提取 PDF 简历文本。
4. 对空文件、无法提取文本或异常文件进行错误处理。
5. 对提取出的文本进行长度控制和基础清洗。
6. 结合大语言模型进行结构化分析，提取技能、项目经历等信息。
7. 将解析结果用于简历展示、岗位匹配和面试上下文构造。

由于 PDF 简历可能存在扫描件、图片型内容、复杂排版和文本顺序异常等情况，系统不将简历解析结果视为完全准确结果。用户仍需要结合页面展示对简历解析内容进行理解和判断。

### 3.1.3 岗位匹配与岗位模板需求

岗位匹配模块用于将用户简历与目标岗位方向关联，为后续模拟面试提供上下文基础。系统需要支持用户选择岗位方向或岗位模板。岗位模板可包含岗位名称、面试方向、系统提示词或提问约束，用于初始化面试会话并控制问题生成范围。

岗位匹配与岗位模板模块需要满足以下需求：

1. 支持用户选择目标岗位方向或岗位模板。
2. 支持系统维护不同岗位模板。
3. 岗位模板可包含面试方向、岗位说明、系统提示词或提问约束。
4. 简历解析结果可用于辅助岗位匹配。
5. 岗位匹配结果用于辅助生成面试问题。
6. 岗位匹配结果不作为用户岗位胜任能力的确定性评价。

### 3.1.4 模拟面试与多轮问答需求

模拟面试模块是系统的核心交互模块。系统根据用户简历、岗位模板和历史消息构造模型调用上下文，并通过大语言模型生成面试追问。为提升问答过程的交互连续性，后端通过 SSE 将模型输出分片返回给前端，由前端逐步展示面试官回复。

模拟面试模块需要满足以下需求：

1. 支持用户基于某份简历和岗位模板创建面试会话。
2. 创建会话时写入系统提示词，作为后续模型调用的上下文约束。
3. 用户每轮提交回答后，后端先保存 `user` 消息。
4. 后端根据简历、岗位模板和历史消息构造上下文。
5. 系统调用用户级模型配置或可用模型配置生成追问。
6. 后端通过 SSE 分片返回 `assistant` 回复。
7. 流式输出完成后，后端保存完整 `assistant` 消息。
8. 前端支持流式展示、请求中止、状态核对和异常恢复。
9. 系统能够基于会话消息为后续报告生成提供完整数据来源。

### 3.1.5 异步报告生成需求

面试结束后，系统需要根据完整对话内容生成面试报告。报告生成通常包含历史消息读取、提示词构造、模型调用、结构化解析和结果落库等步骤，耗时受模型服务和网络条件影响。为避免 `/finish` 请求长时间阻塞，系统通过 RabbitMQ 实现异步报告生成。

异步报告生成模块需要满足以下需求：

1. 用户在面试页面点击结束面试。
2. 前端调用 `/finish` 接口。
3. 后端校验 session 归属和当前状态。
4. 合法会话的 `session.status` 从 `ongoing` 更新为 `generating`。
5. 后端生成 `jobId`，并向前端返回 `generating + jobId`。
6. 后端通过 `RabbitTemplate.convertAndSend(...)` 发布 `ReportJobMessage`。
7. `ReportJobWorker` 通过 `@RabbitListener` 消费 RabbitMQ 消息。
8. 消费者检查 session 是否仍为 `generating`。
9. 重复或过期消息应跳过处理。
10. 报告生成完成后写入 `summary_report`。
11. 会话状态更新为 `finished`。
12. 后端通过 SSE 广播 `report_ready`。
13. 前端收到 `report_ready` 后刷新报告内容。

报告生成链路设计如下：

```text
/finish → generating → RabbitMQ → ReportJobWorker → summary_report → finished → report_ready
```

其中，`/finish` 返回 `generating + jobId` 只表示报告任务进入异步生成链路，不表示报告已经完成。报告是否可查看应以后端完成报告写入、会话状态变为 `finished` 以及前端收到 `report_ready` 事件为准。

### 3.1.6 用户级 BYOK 模型配置需求

不同用户可能使用不同的模型服务或访问凭证。为提高模型接入灵活性，系统支持用户级 OpenAI-compatible BYOK 配置。用户可以配置 endpoint root、API Key 和运行模型，系统在聊天、流式面试和报告生成链路中复用该配置。

用户级 BYOK 模型配置模块需要满足以下需求：

1. 用户可以填写 OpenAI-compatible endpoint root。
2. 用户可以填写 API Key。
3. 系统支持调用模型发现接口获取模型列表。
4. 用户可以选择运行模型并保存配置。
5. API Key 由后端加密保存。
6. 前端查询配置时不回显 API Key 明文。
7. 聊天、流式面试和报告生成链路复用用户级配置。
8. 系统不绑定默认厂商，不推荐具体模型。
9. 系统不宣称兼容所有 OpenAI-compatible 平台。

### 3.1.7 报告展示与数据看板需求

报告展示模块用于向用户呈现面试总结、评分、优势、薄弱点和改进建议。数据看板模块用于展示历史报告结果、能力变化趋势和薄弱点统计，为用户后续训练提供参考。

报告展示与数据看板模块需要满足以下需求：

1. 展示面试总结、评分维度、优势分析、薄弱点和建议。
2. 支持根据报告数据进行图表展示。
3. 支持查看历史面试结果。
4. 支持展示能力变化趋势和薄弱项统计。
5. 报告内容和评分仅作为求职训练辅助反馈。
6. 报告结果不作为用户能力的最终判断。

### 3.1.8 本地验收数据夹具隔离需求

为支持本地开发、人工验收和答辩展示，系统需要具备本地验收数据夹具（dev fixture）能力。local/dev 数据夹具只用于本地验收，Full Docker / prod 默认不加载 data-dev.sql，也不开放 reset API。

本地验收数据夹具模块需要满足以下需求：

1. 支持 local/dev 验收数据重置。
2. 本地验收数据夹具数据不应进入 Full Docker / prod 默认路径。
3. 本地验收数据夹具模式下的状态和数据应具备可重置性。
4. 本地验收数据夹具不应与 Full Docker / prod 配置混用。
5. 本地测试和验收结论不应扩展为公网生产环境结论。

## 3.2 非功能性需求分析

非功能性需求用于约束系统在安全性、可用性、可维护性、可扩展性和边界控制等方面的设计。由于本系统涉及用户简历、模型 API Key 和面试记录等信息，非功能性设计需要同时考虑数据保护、异常处理和工程可维护性。

### 3.2.1 安全性需求

系统需要对用户身份、资源访问和敏感配置进行保护。用户登录后通过 JWT 访问受保护接口，后端根据用户上下文校验资源归属。对于用户上传文件，系统需要进行基础文件安全校验。对于用户模型配置中的 API Key，系统需要后端加密保存，前端不回显明文。

安全性需求包括：

1. 后端通过 JWT 实现接口鉴权。
2. 用户只能访问属于自己的简历、会话、报告和模型配置。
3. 请求处理过程中需要维护当前用户上下文，并在请求结束后清理。
4. API Key 由后端加密保存。
5. 前端查询模型配置时不回显 API Key 明文。
6. 文件上传需要进行类型、大小和 PDF 魔数校验。
7. 未登录、令牌无效或越权访问时，系统应拒绝请求。
8. 异常响应不应直接暴露底层敏感堆栈信息。

### 3.2.2 可用性需求

系统应尽量保证本地和展示环境下的核心功能链路可用。模拟面试过程中，SSE 流式响应可以提升问答过程的交互连续性。前端可通过 `requestAnimationFrame` 缓冲刷新减少频繁渲染风险，通过 `AbortController` 中止活跃请求，通过状态核对和指数退避降低异常路径下的界面状态错乱风险。

可用性需求包括：

1. 面试问答支持流式展示。
2. 报告生成完成后通过 `report_ready` 通知前端刷新。
3. 前端支持中止当前活跃请求。
4. 页面状态应根据后端会话状态进行核对。
5. 网络异常或 SSE 中断时，应通过重试或状态刷新降低界面状态不一致风险。
6. 相关机制仅作为本地、展示和功能链路层面的稳定性设计，不作为大规模长连接验证结论。

### 3.2.3 可维护性需求

系统采用前后端分离架构，前端负责页面展示和交互状态管理，后端负责业务逻辑、数据持久化和中间件集成。后端模块应围绕用户、简历、岗位模板、面试会话、消息、报告、模型配置等对象划分，降低模块之间的耦合。

可维护性需求包括：

1. 前后端职责清晰，接口边界明确。
2. 后端服务按业务模块组织。
3. 数据库对象与业务对象保持对应关系。
4. Redis、RabbitMQ、MySQL 等中间件职责清晰。
5. Redis 用于限流、缓存、评分锁和状态辅助。
6. RabbitMQ 用于报告生成异步任务队列。
7. 模型调用通过统一网关或服务层封装。
8. 异常处理和返回结构保持统一。

### 3.2.4 可扩展性需求

系统需要支持岗位模板扩展、模型服务配置扩展和报告生成流程扩展。岗位模板可扩展不同面试方向；BYOK 模型配置使用户能够配置不同 OpenAI-compatible endpoint；RabbitMQ 使报告生成与 `/finish` 请求解耦，为后续任务状态查询、重试和补偿机制预留空间。

可扩展性需求包括：

1. 岗位模板可根据不同求职方向扩展。
2. 用户级 BYOK 配置可支持不同 endpoint root、API Key 和 model。
3. 报告生成通过 RabbitMQ 与用户请求解耦。
4. 数据看板可基于评分记录和薄弱点记录扩展更多统计维度。
5. 本地验收数据夹具模式应与 Full Docker / prod 配置保持边界，便于后续维护。

上述可扩展性设计主要服务于系统后续迭代，不表示当前系统已经具备复杂部署环境下的性能能力。

### 3.2.5 可靠性边界需求

RabbitMQ 在当前系统中承担报告生成基础异步队列职责，用于将报告任务从用户请求链路中拆分出来。当前设计能够表达报告任务发布、消费者处理和状态更新的基本流程，但仍需要明确可靠性边界。

可靠性边界包括：

1. RabbitMQ 当前用于报告生成异步任务队列。
2. 当前未引入 DLQ、outbox、publisher confirm 和消费并发调优。
3. 系统不声明消息不会丢失。
4. 系统不将当前链路表述为完整可靠消息处理方案。
5. 生产环境下的消息确认、异常补偿和任务恢复仍需后续设计和测试。
6. 本章仅从系统分析与设计角度说明异步链路，不扩展为测试结论。

### 3.2.6 模型输出边界需求

大语言模型输出存在不确定性。系统通过 Structured Output、字段校验、分数范围约束和异常容错降低报告解析失败风险，但不能保证模型输出始终合法。系统生成的报告和评分应作为训练辅助反馈，不应作为用户能力或招聘结果的最终判断。

模型输出边界包括：

1. 模型输出可能存在格式波动。
2. Structured Output 用于降低解析失败风险。
3. 后端需要进行字段校验和异常处理。
4. 分数字段需要进行范围约束。
5. 报告评分和建议仅作为训练参考。
6. 系统不替代真实招聘评价。

## 3.3 数据库设计说明

数据库设计需要支撑用户、简历、岗位模板、面试会话、消息记录、报告、模型配置和统计数据等核心对象。系统采用关系型数据库保存主要业务数据，并通过对象之间的关联关系描述求职训练流程。

### 3.3.1 用户数据设计

`user` 用于保存用户账号、认证相关信息和用户级配置归属关系。用户是系统中简历、面试会话、报告和模型配置的归属主体。系统在访问相关资源时，需要根据当前登录用户校验资源是否属于该用户。

用户数据设计需要满足以下要求：

1. 支持用户注册和登录。
2. 保存账号和认证相关信息。
3. 作为简历、会话、报告和模型配置的数据归属主体。
4. 支持后端根据用户身份进行资源归属校验。

### 3.3.2 简历数据设计

`resume` 用于保存用户上传的简历信息、原始文本和结构化解析结果。一个用户可以拥有多份简历，因此 `resume` 与 `user` 之间为多对一关系。简历数据既用于页面展示，也用于岗位匹配和面试上下文构造。

简历数据设计应包含以下内容：

1. 简历所属用户。
2. 简历文件相关信息。
3. PDF 提取出的原始文本。
4. 结构化解析结果，例如技能和项目经历。
5. 创建和更新时间等基础信息。

### 3.3.3 岗位模板数据设计

`position_template` 用于保存岗位方向、岗位名称、系统提示词或提问约束等信息。岗位模板用于面试会话初始化，也用于控制大语言模型生成问题时的方向和边界。

岗位模板数据设计需要满足以下要求：

1. 保存岗位名称或岗位方向。
2. 保存面试问题生成所需的系统提示词或约束信息。
3. 支持不同岗位方向扩展。
4. 为面试会话创建提供模板基础。

### 3.3.4 面试会话数据设计

`interview_session` 用于保存一次模拟面试的基本信息，包括用户、简历、岗位模板、会话状态和报告相关内容。一个用户可以创建多个面试会话，一份简历也可以对应多次面试训练。

会话状态使用三态设计：

| 状态 | 含义 |
| --- | --- |
| `ongoing` | 面试进行中，用户可以继续提交回答 |
| `generating` | 用户已结束面试，报告任务进入异步生成链路 |
| `finished` | 报告已生成并可查看 |

三态设计用于区分“面试仍在进行”“报告正在生成”和“报告已经完成”三个阶段。用户触发 `/finish` 后，会话从 `ongoing` 更新为 `generating`；报告生成完成后，会话更新为 `finished`。

### 3.3.5 面试消息数据设计

`interview_message` 用于保存一次面试会话中的消息记录。消息角色包括 `system`、`user` 和 `assistant`。其中，`system` 消息通常用于保存岗位模板和面试约束，`user` 消息保存用户回答，`assistant` 消息保存模型生成的面试官回复。

面试消息数据设计需要满足以下要求：

1. 每条消息关联一个面试会话。
2. 使用角色字段区分 `system`、`user` 和 `assistant`。
3. 使用 `seq_num` 或等价顺序字段维护会话内消息顺序。
4. 支持后端构造多轮上下文。
5. 支持报告生成阶段读取完整历史记录。

### 3.3.6 报告数据设计

报告生成结果可保存为与面试会话关联的 `summary_report` 数据。报告内容包括面试总结、评分、优势、薄弱点和建议等信息。报告由 `ReportJobWorker` 消费 RabbitMQ 消息后生成，并在生成完成后写入数据库。

报告数据设计需要满足以下要求：

1. 报告与面试会话关联。
2. 保存总结、评分、优势、薄弱点和建议等内容。
3. 报告写入后，会话状态更新为 `finished`。
4. 报告用于前端展示和数据看板输入。
5. 报告和评分仅作为训练辅助反馈。

### 3.3.7 用户模型配置数据设计

用户模型配置用于保存用户级 OpenAI-compatible endpoint root、运行模型和加密后的 API Key。用户查询配置时，系统不返回 API Key 明文，只返回必要的配置状态和脱敏信息。

用户模型配置数据设计需要满足以下要求：

1. 配置与用户关联。
2. 保存 endpoint root。
3. 保存运行模型。
4. 保存加密后的 API Key。
5. 查询时不返回 API Key 明文。
6. 支持聊天、流式面试和报告生成链路复用。

### 3.3.8 评分、弱点与统计数据设计

系统可通过 `score_history`、`user_weakness` 等辅助数据记录评分历史和薄弱点信息，用于报告展示和数据看板。此类数据主要用于训练反馈和趋势观察，不用于最终能力评价。

评分、弱点与统计数据设计需要满足以下要求：

1. 记录用户或会话对应的评分结果。
2. 记录薄弱点和建议。
3. 支持历史趋势展示。
4. 支持数据看板统计。
5. 不将统计结果作为招聘决策依据。

## 3.4 系统架构说明

系统采用前后端分离架构，整体可划分为前端展示层、后端服务层、业务模块、模型服务层、数据与中间件层以及可选观测层。图 3.3 是系统分层架构图，用于展示前端、后端、业务模块、模型服务、MySQL、Redis、RabbitMQ 和可选观测组件之间的基本关系。RabbitMQ 在图中作为报告生成异步任务队列出现，其具体处理链路在正文中说明，并在第四章实现部分展开。

### 3.4.1 前端展示层

前端展示层采用 Vue 3、TypeScript 和 Vite 构建，主要负责用户交互和页面展示。前端页面包括登录注册、简历上传与管理、岗位选择、模拟面试、报告展示、数据看板和 BYOK 配置等。

前端展示层的主要职责包括：

1. 提供用户登录与路由控制。
2. 支持 PDF 简历上传和解析结果展示。
3. 支持岗位选择和面试会话创建。
4. 展示 SSE 流式返回的面试官回复。
5. 接收 `report_ready` 事件并刷新报告。
6. 展示报告内容和数据看板。
7. 提供用户级 BYOK 配置页面。
8. 在异常情况下进行状态提示、请求中止和状态核对。

### 3.4.2 后端服务层

后端服务层采用 Spring Boot 构建，负责提供 REST API、执行业务逻辑、访问数据库、调用模型服务并集成中间件。后端需要对用户身份、会话状态和资源归属进行校验，以保证业务流程符合设计约束。

后端服务层的主要职责包括：

1. 用户注册、登录和 JWT 鉴权。
2. 简历上传、文件校验和 PDF 文本提取。
3. 简历结构化分析和岗位匹配辅助。
4. 面试会话创建、消息保存和上下文构造。
5. SSE 流式响应和事件通知。
6. `/finish` 状态校验和报告任务发布。
7. RabbitMQ 报告任务消费和报告写入。
8. BYOK 模型配置保存、查询和模型发现。
9. 报告查询和数据看板接口。
10. 异常处理和统一响应封装。

### 3.4.3 数据持久层与中间件层

数据持久层和中间件层用于支撑系统的核心数据保存、缓存、限流和异步任务处理。MySQL 用于保存用户、简历、岗位模板、面试会话、消息、报告、模型配置和统计数据。Redis 用于限流、缓存、评分锁和状态辅助。RabbitMQ 用于报告生成异步任务队列。

各组件职责如下：

| 组件 | 主要职责 |
| --- | --- |
| MySQL | 保存核心业务数据 |
| Redis | 限流、缓存、评分锁、状态辅助 |
| RabbitMQ | 报告生成异步任务队列 |

需明确的是，Redis 不承担报告生成任务队列职责。报告生成任务从 `/finish` 请求中解耦后，由 RabbitMQ 负责消息转交和消费者处理。

### 3.4.4 外部模型服务层

外部模型服务层通过 OpenAI-compatible 调用网关接入用户配置的模型服务。系统支持用户级 BYOK 配置，用户可配置 endpoint root、API Key 和运行模型。聊天、流式面试和报告生成链路均可以读取用户保存的配置并进行调用。

外部模型服务层设计边界包括：

1. 通过统一模型调用网关组织请求。
2. 支持用户级 endpoint root、API Key 和 model。
3. API Key 加密保存，前端不回显明文。
4. 不绑定默认厂商，不推荐具体模型。
5. 不宣称兼容所有 OpenAI-compatible 平台。
6. 对模型输出进行结构化约束、字段校验和异常处理。

### 3.4.5 架构边界说明

当前系统架构服务于本地开发、start-dev 本地验收和 start-docker 部署验证。系统设计中包含 Redis、RabbitMQ、SSE、BYOK 和 Structured Output 等机制，但这些机制在本章中仅作为分析与设计内容，不表示已经完成公网环境性能验证或复杂部署环境可靠性验证。图 3.3 应理解为系统分层和主要数据流示意，其中 RabbitMQ 表示报告生成异步任务队列，不表示系统已经具备 DLQ、outbox、publisher confirm、消费并发调优或生产级消息可靠性保障。

## 3.5 核心业务流程设计

核心业务流程设计用于说明系统主要功能如何在模块之间流转。本节从简历解析与岗位匹配、模拟面试流式问答、异步报告生成和 BYOK 模型配置四个方面进行说明。

### 3.5.1 简历解析与岗位匹配流程

简历解析与岗位匹配流程用于将用户上传的 PDF 简历转换为后续面试可使用的上下文基础。流程如下：

```text
用户上传 PDF 简历
→ 后端校验文件类型、PDF 魔数和文件大小
→ PDFBox 提取文本
→ 文本长度和空文本校验
→ LLM 结构化分析
→ 保存简历结构化结果
→ 岗位模板匹配或岗位方向选择
→ 生成面试上下文基础
```

该流程中，文件校验用于降低异常文件进入解析流程的风险；PDFBox 用于提取 PDF 文本；大语言模型用于辅助识别技能和项目经历等结构化字段。解析结果保存后，可用于岗位匹配和面试上下文构造。

由于 PDF 文件格式复杂，系统不保证所有 PDF 简历都能被正确提取文本。对于扫描件、图片型简历和复杂排版文件，系统需要通过异常处理提示用户。岗位匹配结果用于辅助问题生成，不作为用户能力或岗位胜任程度的最终评价。

### 3.5.2 模拟面试流式问答流程

模拟面试流程用于支持用户与系统进行多轮问答。流程如下：

```text
用户创建会话
→ 系统绑定简历和岗位模板
→ 初始化 system 提示词
→ 用户提交回答
→ 后端保存 user 消息
→ 后端构造上下文
→ 调用用户级模型配置或可用模型配置
→ SSE 分片返回 assistant 回复
→ 前端流式展示
→ 后端保存完整 assistant 消息
→ 等待下一轮回答
```

在该流程中，系统提示词用于约束面试方向，用户消息和模型回复按顺序写入消息表。后端在每轮对话中构造上下文，以保持面试过程的连续性。前端通过流式展示减少用户等待完整回复的时间。

前端可以结合缓冲刷新、请求中止和状态核对等机制提升交互可用性。这些机制用于降低本地和展示环境中页面状态错乱或请求冲突的风险，不表示系统已经完成大规模长连接验证。模型输出仍存在不确定性，系统需要在业务层控制其使用边界。

### 3.5.3 异步报告生成流程

异步报告生成流程是系统结束面试后的关键流程。该流程用于将长耗时报告生成任务从用户请求中解耦。流程如下：

```text
/finish
→ generating
→ RabbitMQ
→ ReportJobWorker
→ summary_report
→ finished
→ report_ready
```

具体流程说明如下：

1. 用户点击结束面试，前端调用 `/finish` 接口。
2. 后端校验 session 归属和当前状态。
3. 若会话合法且处于 `ongoing`，后端将状态更新为 `generating`。
4. 后端生成 `jobId`，并向前端返回 `generating + jobId`。
5. 后端通过 RabbitMQ 发布 `ReportJobMessage`。
6. RabbitMQ 接收报告任务消息。
7. `ReportJobWorker` 通过 `@RabbitListener` 消费任务。
8. 消费者读取 session 并检查状态是否仍为 `generating`。
9. 如果消息重复或 session 状态已变化，则跳过处理。
10. 如果状态有效，则读取面试历史并调用模型生成报告。
11. 后端解析报告并写入 `summary_report`。
12. 会话状态更新为 `finished`。
13. 后端通过 SSE 广播 `report_ready`。
14. 前端收到通知后刷新报告页面。

该流程中，`generating` 是报告生成中的中间状态，用于区分“用户已结束面试但报告尚未完成”和“报告已完成”。RabbitMQ 用于报告任务转交，Redis 不承担该任务队列职责。当前设计未包含 DLQ、outbox、publisher confirm 和消费并发调优，因此不将该链路表述为完整可靠消息处理方案。

### 3.5.4 BYOK 模型配置流程

BYOK 模型配置流程用于支持用户配置自己的 OpenAI-compatible 模型服务。流程如下：

```text
用户填写 endpoint root 和 API Key
→ 调用模型发现接口
→ 返回模型列表
→ 用户选择 model
→ 保存配置
→ API Key 加密保存
→ 前端查询不回显明文
→ 聊天、流式面试和报告生成复用配置
```

具体设计说明如下：

1. 用户在配置页面输入 endpoint root 和 API Key。
2. 系统调用模型发现接口，请求用户配置的 endpoint。
3. 后端返回模型列表供前端展示。
4. 用户选择运行模型并保存配置。
5. 后端加密保存 API Key。
6. 前端查询配置时只显示配置状态或脱敏信息。
7. 后续聊天、流式面试和报告生成读取用户级配置。
8. 当配置不可用时，系统应返回明确错误提示。

BYOK 设计提高了模型接入灵活性，但不同 OpenAI-compatible endpoint 的接口实现可能存在差异。因此，系统不绑定具体厂商，不推荐具体模型，也不宣称兼容所有 OpenAI-compatible 平台。

### 3.5.5 报告展示与数据看板流程

报告展示与数据看板流程用于向用户呈现训练结果。流程如下：

```text
报告生成完成
→ summary_report 写入数据库
→ 会话状态变为 finished
→ 前端收到 report_ready
→ 前端刷新报告
→ 展示总结、评分、优势和薄弱点
→ 数据看板更新历史趋势
```

报告展示页面主要向用户展示面试表现总结、能力评分、优势、薄弱点和改进建议。数据看板可基于历史评分和薄弱点记录展示趋势变化。该流程用于辅助用户复盘训练结果，不用于替代真实招聘评价。

### 3.5.6 本地验收数据夹具流程

本地验收数据夹具流程用于支持本地展示或答辩展示。其设计目标是在 local/dev 路径中提供可展示、可重置的业务流程。

本地验收数据夹具设计要求包括：

1. 本地验收数据夹具与 Full Docker / prod 默认路径隔离。
2. local/dev 可通过 reset 端点重置验收数据。
3. local/dev 验收链路和 Full Docker / prod 路径不应混用。
4. 展示链路用于展示核心功能流程，不代表公网生产环境性能结论。

通过上述流程设计，系统能够在本地和展示环境中完成从简历上传、岗位匹配、模拟面试到异步报告生成和结果展示的主要业务闭环。该设计为第四章系统实现和第五章系统测试提供了结构基础。
