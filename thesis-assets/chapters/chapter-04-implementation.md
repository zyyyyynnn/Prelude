# 第四章 系统实现

## 4.1 系统实现总体说明

本系统围绕“智能模拟面试与简历诊断”场景实现，后端采用 Spring Boot、MyBatis-Plus、MySQL、Redis、RabbitMQ 与大语言模型调用网关，前端采用 Vue 3、TypeScript、Vite、shadcn-vue 等技术栈。系统主要功能包括用户认证、简历解析、岗位匹配、模拟面试、多轮上下文管理、SSE 流式响应、评估报告生成、用户级模型配置以及数据看板展示。

从业务流程看，用户完成登录并上传 PDF 简历后，系统通过文件校验、文本提取和结构化分析形成简历数据。用户选择目标岗位后，系统根据岗位模板、简历内容和历史对话构造面试上下文，并通过大语言模型生成阶段化追问。面试结束后，系统不在 `/finish` 请求中同步等待长耗时报告任务，而是进入异步报告生成链路：后端将会话状态更新为 `generating`，通过 RabbitMQ 发布报告任务，由 `ReportJobWorker` 消费并完成报告生成、结果落库和 SSE 通知。报告完成后，会话状态更新为 `finished`，前端再获取并展示报告内容。

本章重点说明第四章范围内的工程实现，不扩展为系统整体测试结论。涉及 Redis、RabbitMQ、Resilience4j、SSE 重连等内容时，仅从实现机制和本地联调边界进行描述，不将其表述为大规模压力测试或可靠消息投递证明。系统整体架构图用于说明主要分层关系；RabbitMQ 报告任务队列的实现细节在本章报告生成模块中展开。

## 4.2 用户认证与基础设施实现

用户认证模块是系统数据隔离的基础。后端在用户注册阶段对用户名唯一性进行校验，并对用户密码进行加盐哈希存储。登录成功后，系统签发 JWT 令牌，前端在后续请求中携带令牌访问受保护接口。后端通过拦截器解析令牌，并将当前用户标识写入请求上下文，使简历、会话、报告等业务接口能够基于当前用户进行数据归属校验。

为降低业务异常直接暴露底层堆栈信息的风险，后端统一封装接口返回结构，并通过全局异常处理机制将参数错误、权限错误、业务状态错误和外部模型调用异常转换为前端可识别的响应格式。对于依赖当前用户身份的服务逻辑，系统通过请求级上下文传递用户标识，并在请求结束后清理上下文，减少线程复用场景下的状态残留风险。

在基础设施层面，系统使用 MySQL 保存用户、简历、岗位模板、面试会话、消息记录、评分历史和薄弱点等数据。Redis 主要用于限流、缓存和状态辅助，例如 LLM 接口调用限流、评分锁或 dev fixture 模式下的状态辅助。当前版本中，Redis 不再承担报告生成任务队列职责。RabbitMQ 被用于报告生成异步任务队列，使报告生成从用户结束面试的短请求中解耦出来。

## 4.3 简历解析模块实现

简历解析模块负责将用户上传的 PDF 简历转换为后续岗位匹配和模拟面试可使用的结构化数据。系统在上传入口进行文件校验，包括文件是否为空、扩展名、文件大小等基础检查。为避免仅依赖后缀名判断文件类型，后端进一步读取文件头部字节，与 PDF 魔数 `%PDF` 进行比对，从而降低伪造扩展名文件进入解析流程的风险。

在文本提取阶段，系统使用 PDFBox 加载 PDF 文件，并通过 `PDFTextStripper` 提取文本。针对简历中常见的多栏排版和版面顺序问题，系统设置 `setSortByPosition(true)`，使文本提取尽量按照页面空间位置排序。若文件无可提取文本或解析异常，后端返回明确业务错误，而不是继续进入后续模型分析流程。

提取到的原始文本作为大语言模型输入，由系统提示词约束模型返回技能、项目经历等结构化字段。后端对模型返回的 JSON 进行反序列化和合法性校验后，再写入 `resume` 表中的 `parsed_skills`、`parsed_projects` 与 `raw_text` 等字段。对于较长的原始文本，系统在落库前执行长度控制；截断时考虑 Unicode 代理对边界，避免 4 字节字符被截断后造成数据库写入异常。

简历解析结果还用于岗位匹配。系统根据简历中的技能点、项目经历与 `position_template` 表中预置岗位模板进行对齐，为后续面试提问提供岗位方向和上下文依据。该匹配结果用于辅助问题生成，不作为对求职者胜任力的确定性评价。

## 4.4 面试会话模块实现

面试会话模块负责管理一次模拟面试的生命周期。用户选择简历和目标岗位后，后端创建 `interview_session` 记录，将当前用户、简历、岗位模板和目标岗位信息关联起来，并将初始状态设为 `ongoing`。同时，系统从岗位模板中读取 `system_prompt`，作为第一条系统消息写入 `interview_message` 表，使后续每轮模型调用都能携带岗位设定和面试官行为边界。

会话消息采用明细表记录，主要字段包括 `session_id`、`role`、`content`、`seq_num`、`score` 和 `hint`。用户每次提交回答后，后端先将用户消息以 `user` 角色写入数据库，再基于当前会话历史构造模型调用上下文。模型生成完成后，系统再将面试官回复以 `assistant` 角色写入数据库。该设计使实时交互、历史回放和报告生成能够共用同一套消息记录。

为控制长对话带来的上下文规模，系统在构造模型输入时采用滑动窗口策略。当历史消息超过阈值时，系统保留第一条系统提示消息，并选取最近若干轮对话参与本次模型调用。该策略主要用于控制 Token 成本和降低长上下文响应波动，同时尽量保留最近交互语境。

当前 `interview_session.status` 使用三态状态机：

| 状态 | 含义 |
| --- | --- |
| `ongoing` | 面试进行中，用户仍可继续对话 |
| `generating` | 用户已触发 `/finish`，报告任务已发布，等待异步消费或生成中 |
| `finished` | 报告已生成并写入 `summary_report` |

引入 `generating` 中间态，目的是避免报告生成期间前端误判会话已经完成。用户触发 `/finish` 后，系统仅表示面试已结束并进入报告生成链路，不表示最终报告已经可读。待 `ReportJobWorker` 完成报告生成并写入数据库后，状态才切换为 `finished`。

## 4.5 流式模拟面试与前端交互实现

模拟面试模块采用 SSE 实现服务端到前端的流式文本推送。用户提交回答后，后端在完成权限和状态校验后写入用户消息，再调用大语言模型流式接口。模型返回增量文本时，后端通过 `SseEmitter` 逐段发送 `message` 事件，前端接收后实时追加到面试官气泡中。流式输出结束后，后端识别 OpenAI-compatible 协议中的 `[DONE]` 终止标记，并将完整回复写入数据库。

前端为降低高频文本切片带来的响应式更新压力，在面试视图中引入基于 `requestAnimationFrame` 的缓冲刷新机制。前端先将 SSE 接收到的片段写入非响应式缓冲区，再在浏览器下一次重绘周期中批量更新视图。该机制用于减少长文本输出过程中的频繁 DOM 更新风险，但本文不将其表述为固定帧率能力。

在异步交互管理方面，前端使用 `AbortController` 管理当前活跃流式请求。当用户主动结束面试或切换流程时，前端会中止当前网络流，避免多个异步请求同时修改同一会话状态。对于网络异常或 SSE 闪断场景，系统通过前端状态核对和指数退避重试机制降低重复提交和界面状态错乱风险。该机制属于错误恢复设计，尚未扩展为长连接高并发或极端并发测试结论。

前端页面还承担简历管理、岗位选择、面试对话、报告预览和数据看板展示等交互职责。报告展示阶段，前端在 `/finish` 返回 `generating` 后保持等待状态；后端通过 SSE 广播 `report_ready` 事件后，前端再拉取或刷新报告内容，从而避免单次 HTTP 请求长时间阻塞。

## 4.6 评估报告生成模块实现

评估报告生成模块用于在面试结束后，对完整对话内容进行总结、评分和反馈提取。当前实现中，系统没有将长耗时报告生成任务直接阻塞在 `/finish` 请求中，而是通过 RabbitMQ 进行异步解耦。

报告生成链路如下：

```text
/finish → generating → RabbitMQ → ReportJobWorker → summary_report → finished → report_ready
```

用户触发 `/finish` 后，`InterviewServiceImpl.finish(...)` 首先校验会话归属和当前状态。校验通过后，后端将 `interview_session.status` 更新为 `generating`，生成 `jobId`，并通过 `RabbitTemplate.convertAndSend(...)` 向报告任务交换机发布 `ReportJobMessage`。该消息包含 `sessionId`、`userId` 和 `jobId`，供消费者恢复用户上下文并定位目标会话。

`ReportJobWorker` 通过 `@RabbitListener` 监听报告任务队列。消费者收到消息后，会读取目标会话，并检查会话状态是否仍为 `generating`。如果状态已经不是 `generating`，说明该消息可能为重复或过期任务，系统会跳过处理，避免重复调用模型、重复写库或重复广播。若状态有效，消费者继续执行报告生成流程，包括读取面试历史、构造评估提示词、调用模型、解析结构化结果、写入 `summary_report`，并更新会话状态为 `finished`。

报告完成后，后端通过 SSE 广播 `report_ready` 事件，通知前端报告已生成。前端收到事件后刷新报告展示区域。该设计使用户结束面试的请求能够较快返回 `generating` 状态，同时将外部模型长耗时调用放入异步消费者中执行。

Redis 在当前报告生成链路中不再作为任务队列使用，而保留在限流、缓存和状态辅助等轻量场景中。由于当前实现未引入 DLQ、outbox、publisher confirm 或消费并发调优，因此本文仅将 RabbitMQ 表述为报告生成异步任务队列的实现，不将其扩展为完整可靠消息机制证明。

## 4.7 Structured Output 与报告解析实现

报告内容由大语言模型根据面试全程对话生成。为降低自由文本解析的不稳定性，系统在报告生成模块中引入 Structured Output 思路，通过提示词和结构约束要求模型输出符合预期字段的结果。后端对模型返回内容进行反序列化、字段校验和容错处理，减少格式波动对报告展示的影响。

报告结构主要服务于以下展示和分析目标：一是形成面试表现总结，二是提取技术能力、表达能力、逻辑能力等维度评分，三是整理优势与薄弱点，四是为数据看板提供可视化输入。系统会将最终报告写入 `interview_session.summary_report`，并结合评分历史、薄弱点记录等数据支撑后续图表展示。

对于分数字段，系统在落库前进行范围约束和兜底处理，避免异常输出直接影响前端展示。对于结构化字段缺失或格式异常的情况，系统通过异常捕获和局部降级降低报告生成链路中断风险。该机制能够降低解析失败概率，但不能消除大语言模型输出不确定性，因此报告仍应定位为求职训练辅助反馈，不宜作为对用户能力的最终判断。

## 4.8 用户级 BYOK 模型服务配置实现

系统已实现用户级 OpenAI-compatible BYOK 配置能力。用户可以在设置界面填写 endpoint root 与 API Key，后端通过模型发现接口调用 `{root}/models` 获取可用模型列表，前端展示检测结果供用户选择运行模型。用户保存配置后，聊天、流式面试和 RabbitMQ 报告生成链路均可复用该用户级 endpoint、API Key 和 model。

后端接口主要包括模型发现、配置查询、配置保存和配置测试。模型发现接口使用用户本次提交的 endpoint 与 API Key 调用远端模型列表接口，不保存 API Key。配置保存接口将用户选择的 endpoint、运行模型和 API Key 写入用户配置，其中 API Key 在后端加密保存。配置查询接口仅返回是否已配置密钥和脱敏状态，前端不回显 API Key 明文。

在模型调用阶段，系统将用户保存的配置作为运行参数传入 LLM 调用网关。模型名称仅表示用户选择的运行模型，不作为系统默认推荐，也不绑定某个固定厂商。该能力用于支持符合 OpenAI-compatible 接口形式的用户级模型配置，不表示系统兼容所有平台，也不表示已完成公网性能验证。

## 4.9 Redis、Resilience4j 与稳定性机制实现

系统在稳定性设计上采用多种机制进行分层处理。Redis 主要用于高成本模型接口的限流、缓存和状态辅助。后端在 LLM 调用入口通过 Redis 与 Lua 脚本实现细粒度限流，减少单一用户短时间内频繁调用模型接口造成的资源消耗。该机制属于接口保护设计，目前仅可作为实现机制描述，不写成大流量压测结论。

在外部模型调用链路中，系统引入 Resilience4j 熔断机制。LLM 调用网关可根据异常率或超时情况进入熔断状态，并在配置允许时切换到备选通道。该设计用于降低外部模型服务异常对主流程的影响，但其自动切换效果、极端拥塞下的触发行为和恢复时间仍需专项测试数据支撑。

SSE 流式通信方面，系统在后端使用 `SseEmitter` 推送增量内容，在前端通过流式读取、缓冲渲染、请求互斥和状态核对机制处理网络异常。报告生成完成事件也通过 SSE 广播，使前端能够在异步报告完成后刷新界面。上述机制主要用于提升本地和展示环境下的链路韧性与错误恢复能力，不作为大规模长连接能力证明。

此外，系统保留 dev fixture 本地验收数据夹具能力，使展示环境与 Full Docker / prod 配置在数据、端口和配置上保持边界。Docker Compose 运行模式可管理 MySQL、Redis、RabbitMQ、后端和前端服务；本地运行模式则依赖本机或本机可达的中间件。两种运行方式应避免混用，以减少中间件连接来源不明确带来的排查成本。

综上，本章实现部分重点说明系统从简历解析、岗位匹配、流式面试到异步报告生成的工程闭环。对于 RabbitMQ、Redis、Resilience4j、SSE 和 BYOK 等机制，本文均限定在已有代码和本地联调证据范围内表述，不扩大为公网性能、生产部署或可靠投递层面的结论。
