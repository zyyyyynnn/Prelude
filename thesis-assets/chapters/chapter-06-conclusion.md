# 第六章 总结与展望

## 6.1 工作总结

本文围绕“智能模拟面试与简历诊断系统”的设计与实现展开研究，完成了系统需求分析、总体设计、功能实现和系统测试等工作。系统面向求职训练场景，结合简历解析、岗位匹配、多轮模拟面试和评估报告生成等功能，为用户提供从简历上传到面试反馈的完整使用流程。

在系统功能方面，本文完成了用户认证、简历上传与解析、岗位选择、模拟面试、多轮问答、报告生成和数据看板等核心模块。用户登录后可以上传 PDF 简历，系统对文件进行基础校验和文本提取，并结合大语言模型对简历中的技能、项目经历等信息进行结构化分析。岗位匹配模块根据简历内容和岗位模板辅助确定面试方向，为后续提问生成提供上下文基础。

在系统实现方面，后端采用 Spring Boot、MyBatis-Plus、MySQL、Redis、RabbitMQ 和 LLM 调用网关完成主要业务逻辑。MySQL 用于保存用户、简历、岗位模板、面试会话、消息记录、报告和统计数据；Redis 用于限流、缓存、评分锁和展示状态辅助等场景，不再承担报告生成任务队列职责；RabbitMQ 用于报告生成异步任务队列。前端采用 Vue 3、TypeScript、Vite 等技术实现登录、简历管理、面试交互、报告展示和数据看板等页面。

在模拟面试模块中，系统能够根据岗位模板、简历内容和历史对话构造上下文，支持基于大语言模型的多轮问答。后端通过 SSE 向前端分片推送模型输出，前端结合缓冲刷新、请求中止和状态核对机制完成流式交互展示。该设计改善了面试问答的交互连续性，也为报告生成提供了完整的历史消息记录。

在报告生成模块中，系统将报告生成任务从 `/finish` 请求中解耦。用户结束面试后，应用用例 `FinishInterview` 将会话状态更新为 `generating`，并经 `JobSchedulerPort` 发布报告任务，由 `RabbitJobScheduler` 适配 RabbitMQ。`ReportJobWorker` 通过 `@RabbitListener` 监听队列并转交 `ReportGenerateHandler` 完成报告生成、`summary_report` 写入、会话状态更新为 `finished`，并通过 SSE 广播 `report_ready` 事件。报告生成链路为：

```text
/finish → FinishInterview → generating → JobSchedulerPort → RabbitJobScheduler → ReportJobWorker → ReportGenerateHandler → summary_report → finished → report_ready
```

该链路使 `/finish` 返回 `generating + jobId`，表示报告任务已进入异步生成链路，而不是表示报告已经完成。为了配合该流程，`interview_session.status` 使用 `ongoing / generating / finished` 三态表示面试进行中、报告生成中和报告已完成。

在模型配置方面，系统实现了用户级 OpenAI-compatible BYOK 配置能力。用户可以填写 endpoint root、API Key 和运行模型，系统通过模型发现接口获取可用模型列表，并在配置保存后供聊天、流式面试和 RabbitMQ 报告生成链路复用。API Key 由后端加密保存，前端不回显明文。该能力不绑定默认厂商，也不将某个具体模型作为系统推荐模型。

在报告解析方面，系统引入 Structured Output 和字段校验机制，用于降低大语言模型自由文本输出导致的解析失败风险。后端对模型返回内容进行反序列化、字段校验和分数范围处理，使报告内容能够相对稳定地用于页面展示和数据看板统计。但该机制仅用于降低解析风险，不表示模型输出始终合法，也不表示评分结果具有最终判断效力。

在系统测试方面，第五章围绕本地开发环境、start-dev 本地验收环境和 start-docker 部署验证环境，对核心功能闭环、RabbitMQ 异步报告链路、BYOK 基础链路、SSE 基础交互、Redis 状态辅助和权限隔离进行了验证。测试结果表明，系统主要功能能够在本地环境下按预期完成，报告异步生成链路、用户级模型配置链路和基础数据隔离机制具备基础可用性。

## 6.2 系统不足

尽管系统已经完成主要功能闭环，但受开发周期、测试环境和外部模型服务差异影响，当前系统仍存在以下不足。

第一，大语言模型输出具有不确定性。系统生成的面试追问、报告总结和评分建议依赖模型能力、提示词设计、简历质量和上下文内容。即使引入 Structured Output 和字段校验机制，也只能降低解析失败风险，不能保证模型输出完全符合预期。因此，系统报告应作为求职训练辅助反馈，用户需要结合自身情况理性参考。

第二，当前系统测试主要在本地开发环境、start-dev 本地验收环境和 start-docker 部署验证环境中完成，尚未进行公网大规模并发压测。第五章测试能够说明核心功能链路在本地环境下可用，但不能代表复杂网络环境、长时间运行环境或多用户高负载环境下的性能表现。

第三，RabbitMQ 当前仅完成报告生成异步任务队列集成。系统已经能够通过 RabbitMQ 发布和消费报告任务，但尚未引入 DLQ、outbox、publisher confirm、消费并发调优等可靠消息增强机制。因此，当前实现适合作为本课题中的异步解耦方案说明，后续若面向更复杂部署环境，还需要继续补充消息确认、异常补偿和任务恢复机制。

第四，SSE 流式交互尚未进行大规模长连接稳定性测试。当前系统实现了 SSE 分片返回、前端缓冲渲染、请求中止、状态核对和指数退避等机制，能够支撑本地功能链路验证。但在多用户并发、复杂网络抖动、浏览器长时间挂起等场景下，连接恢复、重复提交控制和状态一致性仍需进一步测试。

第五，BYOK 兼容性受不同 OpenAI-compatible endpoint 实现差异影响。不同平台在模型发现接口、鉴权方式、错误返回格式、流式输出格式和上下文长度限制等方面可能存在差别。当前系统已经支持用户级 endpoint、API Key 和 model 配置，但不能据此说明系统兼容所有 OpenAI-compatible 平台。

第六，简历解析能力仍存在边界。系统能够对常规 PDF 简历进行文件校验、文本提取和结构化分析，但对于扫描件、图片型简历、复杂排版 PDF、加密 PDF 或文本顺序异常的文件，解析效果可能受到影响。岗位匹配结果也主要用于辅助提问生成，不宜作为用户能力或岗位胜任程度的确定性评价。

第七，当前自动化测试体系仍有完善空间。现有测试以基础单元测试、本地接口验证和功能链路验证为主，端到端回归测试、异常路径覆盖、前端交互自动化和多环境持续集成验证仍需进一步补充。

## 6.3 后续展望

针对当前系统的不足，后续可从以下方向继续完善。

第一，补充端到端自动化测试。后续可围绕登录、简历上传、岗位选择、模拟面试、报告生成和数据看板展示等主流程建立自动化回归测试，减少功能迭代中引入回归问题的风险。同时，可增加异常路径用例，例如无效文件上传、未登录访问、报告任务重复触发和模型接口异常返回等场景。

第二，增强 RabbitMQ 报告任务链路的可靠性设计。后续可结合系统实际部署需求，引入 DLQ、publisher confirm、outbox 或任务补偿机制，对消息发布失败、消费者异常退出、任务重复消费和长时间未完成任务进行更完整的处理。对于报告生成这类耗时任务，还可增加任务状态查询接口和失败重试记录，以提升可观测性和可恢复性。

第三，补充专项压力测试。后续可针对 SSE 长连接、Redis 限流、报告任务队列和数据库读写路径设计独立测试方案，评估系统在多用户访问、连续对话、任务堆积和异常网络条件下的表现。相关测试应记录明确的测试环境、并发规模、样本数量和观测指标，避免将机制说明替代测试结论。

第四，优化 BYOK 兼容性。后续可针对不同 OpenAI-compatible endpoint 的返回格式差异，完善模型发现错误提示、模型列表缓存、鉴权失败提示、流式输出异常处理和配置测试结果说明。对于用户配置错误、模型不存在、endpoint 不可达等场景，可以提供更清晰的前端反馈，降低用户排查成本。

第五，提升简历解析能力。后续可以补充 OCR 或外部文档解析能力，以支持扫描件和图片型简历；也可以针对复杂表格、多栏排版和中英文混排简历优化文本清洗策略。同时，应加强字段校验和人工可编辑能力，使用户能够在模型解析不准确时手动修正简历结构化结果。

第六，改进报告评价体系。后续可以引入更多样本进行人工校验，对模型评分与人工评价之间的一致性进行分析，并根据评价结果调整提示词、评分维度和报告结构。对于关键评分项，可以增加解释性说明，帮助用户理解分数来源，降低单次模型输出偏差对用户判断的影响。

第七，探索更丰富的交互形式。在完成当前文本面试主流程稳定性的基础上，后续研究可继续探索语音交互、多模态面试辅助或多模型协同评价等方向。例如，可在远期版本中研究语音识别、语音合成、视频面试辅助和多模型交叉评价机制。但这些方向应建立在充分测试和明确应用边界之上，不能等同于当前系统已经具备的能力。

综上，本文完成了智能模拟面试与简历诊断系统的主要设计、实现和本地测试工作。系统能够支撑简历解析、岗位匹配、流式模拟面试、异步报告生成和用户级模型配置等核心流程。后续工作应重点围绕测试体系完善、异步任务可靠性增强、模型兼容性优化和报告评价质量提升展开，使系统在功能完整性、稳定性和可维护性方面进一步改进。
