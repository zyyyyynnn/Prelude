# 第五章 系统测试

## 5.1 测试目标与方法

系统测试阶段的目标是验证“智能模拟面试与简历诊断系统”在本地开发与展示环境下的核心业务流程是否能够按设计完成。第五章的测试范围定位为系统功能测试与本地集成验证，不等同于上线运维报告，也不扩展为公网环境下的大规模性能测试。

本章测试目标包括以下内容：

1. 验证用户注册、登录、JWT 鉴权、简历上传、岗位选择、模拟面试、报告生成与数据看板展示等核心功能是否能够形成闭环。
2. 验证报告生成链路是否能够在用户触发 `/finish` 后进入 `generating` 状态，并通过 RabbitMQ 异步任务队列完成报告生成。
3. 验证 `interview_session.status` 的三态流转是否符合 `ongoing / generating / finished` 的设计要求。
4. 验证用户级 OpenAI-compatible BYOK 配置是否支持 endpoint root、API Key、模型发现、配置保存、配置测试以及后续链路复用。
5. 验证 SSE 流式响应、前端缓冲渲染、请求中止、状态核对与异常恢复机制在基本交互流程中的可用性。
6. 验证 Redis 在限流、缓存和状态辅助场景中的基础可用性，并确认其不再承担报告任务队列职责。
7. 验证权限隔离、异常输入处理和 dev fixture 本地验收数据夹具机制是否符合系统设计边界。

测试方法以功能测试、接口测试、本地集成测试、异常路径测试和最小自动化测试为主。对于需要真实高并发、复杂网络抖动或生产消息可靠性验证的内容，本章仅说明当前机制和已完成的基础验证，不将其写成已经通过的专项测试结论。

## 5.6 RabbitMQ 异步报告生成链路测试

报告生成链路是第五章测试重点。第四章已明确，当前系统不再使用 Redis 承担报告任务队列，而是通过 RabbitMQ 实现报告生成异步任务队列。测试目标是验证用户结束面试后，系统能否从 `/finish` 进入 `generating`，并最终由消费者生成报告、写入数据库、广播完成事件。

报告生成链路如下：

```text
/finish → generating → RabbitMQ → ReportJobWorker → summary_report → finished → report_ready
```

表 5.5 RabbitMQ 异步报告生成链路测试表

| 测试步骤 | 测试内容 | 预期结果 | 结论 |
| --- | --- | --- | --- |
| 结束面试 | 用户点击结束面试按钮 | 前端调用 `/finish` 接口 | 符合预期 |
| 会话校验 | 后端校验 session 归属和状态 | 非当前用户或非法状态请求被拒绝 | 符合预期 |
| 状态更新 | `/finish` 通过校验后更新会话 | `interview_session.status` 变为 `generating` | 符合预期 |
| 任务编号 | 后端生成 `jobId` | `/finish` 返回 `generating` 和 `jobId` | 符合预期 |
| 消息发布 | 后端通过 `RabbitTemplate.convertAndSend(...)` 发布任务 | RabbitMQ 接收 `ReportJobMessage` | 符合预期 |
| 消息消费 | `ReportJobWorker` 监听队列 | `@RabbitListener` 能够接收报告任务 | 符合预期 |
| 状态守卫 | 消费者检查 session 是否仍为 `generating` | 重复或过期消息被跳过 | 符合预期 |
| 报告生成 | 消费者读取会话历史并调用模型 | 报告内容生成后写入 `summary_report` | 符合预期 |
| 状态完成 | 报告写入后更新会话状态 | `interview_session.status` 变为 `finished` | 符合预期 |
| 完成通知 | 后端通过 SSE 广播 `report_ready` | 前端收到事件后刷新报告 | 符合预期 |

本地 Docker Compose 环境中的链路核对显示，`/finish` 能够返回 `generating` 和 `jobId`，后端日志能够观察到任务发布、消费者接收和 `report_ready` 广播，数据库中的会话最终能够进入 `finished` 状态，且 `summary_report` 有内容写入。RabbitMQ 队列状态也可用于辅助观察消息是否被消费完成。

需说明的是，上述测试验证的是本地 RabbitMQ 基础链路可用，不构成生产环境可靠投递能力证明，也不证明消息不会丢失。当前实现未覆盖 DLQ、outbox、publisher confirm、消费并发调优等可靠消息增强机制，也未进行公网高并发或大规模任务堆积场景测试。

报告解析测试用于验证大语言模型返回的报告内容能否被后端按预期解析、校验并展示。当前系统通过 Structured Output 约束报告输出结构，并在后端进行反序列化、字段校验和分数范围处理，以降低自由文本解析失败的风险。

测试结果表明，Structured Output 与后端校验机制能够降低报告解析失败对页面展示的影响。但模型输出仍具有不确定性，该机制不能保证模型输出始终合法，也不能保证所有评分和建议都具备绝对客观性。因此，系统生成的报告应定位为求职训练辅助反馈，不作为求职者能力的最终判断。

## 5.10 权限与数据隔离测试

权限与数据隔离测试用于验证系统是否能够限制未授权访问，并保证不同用户之间的简历、会话和报告数据不发生越权读取。测试同时覆盖 JWT 缺失、JWT 失效、跨用户访问和 dev fixture 本地验收隔离等场景。

表 5.9 权限与数据隔离测试表

| 测试项 | 测试内容 | 预期结果 | 结论 |
| --- | --- | --- | --- |
| 未登录访问 | 未携带 JWT 访问受保护接口 | 后端拒绝请求，前端引导登录 | 符合预期 |
| 无效令牌访问 | 携带无效或过期 JWT 请求接口 | 请求被拒绝，不返回业务数据 | 符合预期 |
| 用户简历隔离 | 用户访问其他账号简历 | 系统拒绝越权访问 | 符合预期 |
| 会话隔离 | 用户访问其他账号面试会话 | 系统根据当前用户校验归属 | 符合预期 |
| 报告隔离 | 用户访问其他账号报告 | 系统拒绝返回非本人报告 | 符合预期 |
| 前端登录态处理 | JWT 缺失或失效时继续操作 | 前端清理状态并跳转登录页 | 符合预期 |
| 用户上下文清理 | 请求结束后清理用户上下文 | 降低线程复用导致上下文残留风险 | 符合预期 |
| dev fixture 数据夹具边界 | dev fixture 开启时执行数据重置 | local/dev 可重置验收数据，Full Docker / prod 默认路径不加载该夹具 | 符合预期 |

测试结果表明，系统能够在基础接口层面对用户身份和资源归属进行校验，未登录或越权请求无法直接获取受保护数据。dev fixture 本地验收辅助能力能够支持本地展示和测试，但不应与 Full Docker / prod 配置混用。

## 5.11 测试结果与问题分析

综合上述测试结果，系统在本地开发环境和展示环境下完成了核心功能链路验证。用户认证、简历上传、岗位选择、模拟面试、报告生成和数据看板等主要功能能够按预期运行。SSE 流式交互能够完成基本分片返回和前端展示，前端缓冲渲染、请求中止和状态核对机制能够降低部分异常路径下的界面状态错乱风险。

RabbitMQ 异步报告生成链路在本地联调中可用。用户点击结束面试后，`/finish` 能够返回 `generating` 和 `jobId`，报告任务通过 RabbitMQ 发布，`ReportJobWorker` 能够消费任务并生成报告，报告内容写入 `summary_report` 后，会话状态更新为 `finished`，后端通过 SSE 广播 `report_ready`，前端据此刷新报告展示。该链路与第四章实现口径保持一致。

BYOK 用户级模型配置链路也完成了基础功能验证。用户能够输入 endpoint root 和 API Key，系统能够调用模型发现接口获取模型列表，保存用户选择的运行模型，并在聊天、流式面试和 RabbitMQ 报告生成链路中复用该用户级配置。API Key 由后端加密保存，前端不回显明文。

Redis 相关测试表明，其限流、缓存和状态辅助机制具备基础可用性。当前系统中 Redis 不再承担报告任务队列职责，报告异步任务由 RabbitMQ 承担。Structured Output 与报告解析测试表明，系统能够对模型输出进行结构化解析、字段校验和分数范围处理，但模型输出仍存在不确定性，报告结果仍应作为训练辅助反馈使用。

当前测试仍存在以下限制：

1. 未进行公网大规模并发压测，不能证明系统在高并发访问下的吞吐能力和响应稳定性。
2. 未验证生产环境下的消息可靠投递能力，也未覆盖消息丢失、重复投递、消费者异常退出后的完整恢复策略。
3. 当前 RabbitMQ 链路未引入 DLQ、outbox、publisher confirm 和消费并发调优，后续如进入更复杂部署环境，应补充相关机制和测试。
4. SSE 流式交互尚未进行大规模长连接稳定性测试，异常重连、丢包率和重复提交概率仍需专项验证。
5. BYOK 兼容性受第三方 endpoint 实现差异影响，不能据此推断系统兼容所有 OpenAI-compatible 平台。
6. 模型生成报告的内容质量和评分合理性受模型能力、提示词和上下文质量影响，尚需结合更多样本进行评估。
7. 当前自动化测试以基础单元测试和构建验证为主，尚未形成完整的端到端回归测试体系。

后续工作可从四个方向补充：一是增加端到端自动化回归测试，覆盖登录、简历上传、面试和报告生成主流程；二是针对 SSE 长连接、RabbitMQ 消费和 Redis 限流补充专项压力测试；三是引入消息可靠性测试，覆盖重复消息、消费者异常、队列堆积和恢复路径；四是扩大 BYOK endpoint 样本，验证不同 OpenAI-compatible 服务在模型发现、流式输出和错误返回格式上的兼容性。
