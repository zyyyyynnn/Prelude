# 答辩正式讲稿（5-8 分钟版）

各位老师好，我的毕业设计题目是《基于 LLM 的模拟面试与简历诊断系统》。本系统面向求职准备场景，目标是把简历解析、岗位匹配、阶段化模拟面试、评估报告、历史回放和能力分析串成一条完整的训练闭环。

首先是选题背景。传统面试准备通常依赖人工辅导或简单题库，反馈成本较高，也难以持续记录每次训练中的表现变化。随着大语言模型的发展，系统可以根据简历、岗位和历史回答生成更接近真实面试的追问，并在结束后给出结构化反馈。因此，本课题希望实现一个可本地运行、可演示、可复盘的模拟面试系统。

系统总体采用前后端分离架构。前端使用 Vue 3、TypeScript 和 Element Plus，负责登录、简历管理、面试工作台、数据看板和设置页面。后端使用 Spring Boot，提供认证、简历解析、面试会话、流式问答、报告生成和能力分析接口。数据层使用 MySQL 保存用户、简历、会话、消息、阶段、评分和薄弱点，Redis 用于限流和异步任务队列。外部能力主要包括 PDFBox 文本提取和多 LLM Provider 调用（DeepSeek、OpenAI、Anthropic）。同时，为了保证答辩演示稳定，我设计了 Demo Twin 模式，它使用独立端口、独立数据库和可重置的演示数据，不依赖公网模型也不污染真实模式。

在业务流程上，用户登录后可以上传 PDF 简历，系统通过 PDFBox 提取文本并结合岗位模板创建面试会话。面试过程按照破冰、技术、深挖和收尾阶段推进。用户提交回答后，前端通过 `fetch + ReadableStream` 接收后端 SSE 流式内容，页面可以逐步展示模型回复。当会话超过 15 轮时，系统自动触发 Sliding Window 摘要压缩，将早期对话浓缩为摘要，保证上下文长度可控。面试结束后，系统通过 Structured Output 机制生成 JSON 结构化评分与 Markdown 评估报告，并支持在数据看板中查看能力雷达图、评分趋势和薄弱点统计。

本系统的核心实现主要有四点。第一是 Structured Output 报告解析。旧版方案使用正则提取评分，容易因模型输出格式波动而失败；新版采用 Jackson 反序列化配合 clampScore 安全降级，大幅降低了格式幻觉导致解析失败的概率。第二是流式面试链路。后端将模型返回内容按片段通过 SSE 推送给前端，前端通过 `requestAnimationFrame` 缓冲渲染，减少流式展示时的视觉卡顿。第三是高可用机制。系统引入 Redis Lua 令牌桶限流、Resilience4j 熔断降级和 SSE 指数退避重连，当 Provider 超时或网络中断时自动切换备用通道。第四是异步报告生成。`/finish` 接口快速返回状态，报告生成任务通过 RabbitMQ 队列异步解耦。`InterviewServiceImpl.finish(...)` 将 session 状态置为 `generating` 并通过 `RabbitTemplate.convertAndSend` 发布 `ReportJobMessage`；`ReportJobWorker` 通过 `@RabbitListener` 消费队列，完成后 SSE 推送 `report_ready` 事件；Redis 回归限流、缓存和状态辅助职责。本次验证为本地 Docker Compose 环境下的基础链路联调，不等同于公网高并发压测，也不证明生产级可靠投递或消息绝不丢失。

测试方面，我完成了后端单元测试（14 个测试全绿）、前端生产构建和 Demo Twin 业务测试。业务测试覆盖 `TC-01` 到 `TC-09`，包括合法 PDF 上传、非 PDF 拦截、面试初始化、流式对话、报告生成、空输入、未登录访问、越权访问和 SSE 异常恢复，结果全部通过。需要说明的是，当前测试数据代表 Demo Twin 本机业务闭环与 2026-05-27 真实 API 补充测试，不代表真实公网 LLM 服务的响应性能；公网模型性能后续需要在真实网络和真实 API Key 条件下单独采集。语音交互和部分并发压测指标仍为待实测状态。

开发过程中也遇到了一些真实问题。第一个是 Demo 登录时报 `ECONNREFUSED`。排查后发现是前端缺少 `.env.demo`，导致 Vite 代理默认指向真实后端端口，最终通过补齐 Demo 环境文件并在启动脚本准备阶段校验端口和代理目标解决。第二个是后端启动时 MySQL 连接失败。这个问题不是业务代码本身错误，而是数据库服务未就绪导致 Hikari 初始化失败。后续我把数据库连接、端口和关键密钥检查前置到启动脚本中，让错误更早暴露。

总结来说，本课题完成了一个可运行的模拟面试系统，重点体现了完整业务闭环、SSE 流式体验、Structured Output 结构化评分、Redis 限流与 Resilience4j 熔断、异步报告生成、Docker 容器化编排和证据化测试。当前不足是语音交互和部分性能指标仍为待实测状态，也没有做大规模并发压测。后续可以继续补充语音真实链路测试、批量测试和部署监控。

我的汇报结束，请各位老师批评指正。
